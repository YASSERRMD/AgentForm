# ADR-0015: Agno adapter — a seventh target added post-v1

## Status

Accepted

## Context

`temp/instruction.md`'s 12-phase plan shipped exactly six target framework adapters (OpenAI Agents SDK, LangGraph, Microsoft Agent Framework, Google ADK, AutoGen, CrewAI) and concluded at Phase 12. This ADR covers a seventh, added afterward at explicit user request: [Agno](https://github.com/agno-agi/agno), a Python agent/workflow framework. Unlike every prior adapter phase, there was no existing spec section to implement against — the target framework itself, its real API surface, and how it maps onto Agentform's IR all had to be researched from scratch, matching the same "verify against the real installed package, never guess" discipline every other adapter in this codebase already follows.

Agno turned out to be a better structural match for Agentform's own workflow model than any adapter before LangGraph: its `agno.workflow` module (v2.8.0) exposes `Step`/`Steps`/`Loop`/`Parallel`/`Condition`/`Router`/`HumanReview` as composable Python objects — a near-direct vocabulary match to Agentform's own `agent`/`tool`/`humanApproval`/`loop`/`parallel`/`condition`/`router`/`terminate`/`subworkflow`/`transform`/`delay` node types, verified by actually constructing every one of those shapes against the real installed package (no network calls, construction only) before any code generation logic was written.

## Decision

- **Added as a new adapter package (`@agentform/adapter-agno`), not a modification of any existing one.** `frameworkTargetSchema` (`packages/schema/src/runtime.ts`) gains `'agno'` as a seventh enum value, additive and non-breaking — every existing specification file with a `runtime.target` of one of the original six continues to validate identically.
- **Human approval maps to a genuinely blocking gate, not an emulation.** `Step(requires_confirmation=True)` is Agno's own real human-in-the-loop mechanism (verified against the installed package) — this adapter is the only one besides LangGraph where `humanApproval` is `supported` rather than `unsupported`/`emulated`, and unlike LangGraph's `interrupt()`-based stub, Agno's mechanism needed no manual "TODO: act on the human decision" scaffolding to make it real. The same `requires_confirmation=True` mechanism is also applied at the tool level (`@tool(requires_confirmation=True)`) for any `sideEffect: destructive` tool or `humanApproval`-typed tool — the same structural blast-radius reduction `AF004` exists to enforce.
- **`role`+`description` map into Agno's `description` field, not its own `role` field.** Verified against the installed package's source (`agno/agent/agent.py`): Agno's `Agent.role` specifically means "this agent's role _in a Team_," populated only when the agent is a Team member — using it for Agentform's general-purpose `role` field (required on every agent, Team or not) would be a category error, not a translation, since this adapter doesn't generate `Team` objects.
- **Workflow generation resolves the general-graph-vs-step-sequence mismatch with a cycle-safe depth-first walk from the entrypoint, not a topological sort.** Agentform's workflow model is a general graph (cycles allowed only through `loop` nodes); Agno's is fundamentally an ordered step sequence with structured nesting. A global topological sort (Kahn's algorithm, `@agentform/core`'s `topologicalSort`) was tried first and found — via this adapter's own test suite, not by inspection — to fail completely whenever the entrypoint itself sits inside a loop's cycle: since no node in that cycle ever reaches in-degree zero, _every_ node in the graph is reported unplaceable, not just the loop's own back-edge, producing an empty `steps=[]`. Switching to a pre-order DFS walk starting from the known-good entrypoint sidesteps this: a node is visited once, a revisit through a genuine cycle-closing back-edge is simply skipped (using the same three-color `'visiting'`/`'done'` state `@agentform/core`'s own `findCycle` already uses, to correctly distinguish a real back-edge — the target is a current DFS-stack ancestor — from ordinary DAG convergence, where two independent paths simply reach the same downstream node and neither is a cycle at all).
- **A `parallel`/`router`/`condition` node's nested children are resolved from real IR data, never inferred.** `parallel.branches` is a direct schema field naming the branch node IDs; `router`/`condition`'s choices/then/else come from that node's own outgoing edges. This adapter never attempts to reconstruct "what's inside" an ambiguous graph region — where the mapping is unambiguous (branches, direct outgoing edges), it's generated faithfully; where it would require guessing (a `loop`'s body, a `router`/`condition`'s actual selection logic), the generated code is an honest `NotImplementedError` stub with a TODO, the same discipline every other adapter's conditional-routing code already follows for `when`/`expression` text (Agentform has no expression evaluator anywhere in the codebase, by design).
- **`fastapi` is pinned as a direct dependency of every generated project.** A real, verified finding: `agno.workflow`'s own `__init__.py` unconditionally imports `agno.workflow.remote.RemoteWorkflow`, which requires `fastapi` — confirmed directly (`import agno.workflow` raises `ModuleNotFoundError` without it) — even though `agno`'s own package metadata marks `fastapi` as belonging to the `os`/`dev`/`demo` extras, not a base dependency. Trusting that metadata would produce a generated project that fails to import on a clean `pip install agno`.
- **`join` and `event` node types remain unsupported.** `join`'s semantics belong to the `Parallel` construct that fans out, not to a standalone downstream node a graph walk could bind to; `event` (waiting on an external trigger) has no synchronous `Step` equivalent to translate to honestly. Both are `unsupported` in `validateAgnoCompatibility`, blocking compilation with `AGF5001` rather than silently dropping the node.
- **Agent delegation is `partial`, not `supported`.** This adapter generates plain `Agent` objects wired into `Workflow` `Step`s, not Agno's own real multi-agent delegation construct (`agno.team.Team(members=[...], mode=TeamMode.coordinate)`, verified to exist and construct correctly). Binding `agent.delegation.allowedAgents` to a `Team`'s member set is future adapter work, not attempted here — generating a plain `Agent` is still fully correct for every agent that doesn't declare delegation.

## Alternatives considered

- **Attempting full graph-region reconstruction for `loop`/`router`/`condition` bodies** (inferring which downstream nodes belong "inside" a loop, or serve as a router's implicit continuation): rejected — this is a well-known hard problem (structured-control-flow recovery from a control-flow graph) that risks silently-wrong generated code for graph shapes not covered by this adapter's own tests, which is a worse failure mode than an honest, clearly-marked stub.
- **A global topological sort over the whole workflow graph**: tried and rejected after this adapter's own test suite caught it producing an empty step sequence whenever the entrypoint sits inside a cycle — see the Decision section above. Kept as a cautionary example in `generate-workflow.ts`'s own doc comments rather than silently discarded, since the failure mode is subtle enough to recur if someone "simplifies" the traversal later.
- **Mapping `role` to Agno's own `role` field** (the more superficially obvious 1:1 name match): rejected once the installed package's source revealed `role` is Team-membership-specific, not general-purpose — using it here would have been confidently wrong, not just imprecise.
- **Treating `tool`-type workflow nodes the same as agent-invoked tools**: rejected — Agno's tools are designed to be invoked by an agent's own reasoning; there's no "run this tool directly as a step, no agent involved" primitive to translate to faithfully, so a `tool` node becomes a documented `Step(executor=<stub>)` rather than a fabricated direct-invocation pattern.

## Consequences

- `ADAPTER_REGISTRY` (`apps/cli/src/lib/generate-artifacts.ts`), `VALID_TARGETS` (`apps/cli/src/commands/init.ts`), and the cross-target test (`apps/cli/src/cross-target.test.ts`) all gained an `agno` entry — every place that previously enumerated "the six targets" now enumerates seven, found and fixed by grepping for the prior hardcoded lists rather than assuming completeness.
- `@agentform/adapter-agno` depends on `@agentform/core` for its DFS-walk and `parseDurationMs` (for `delay` nodes) — a new dependency edge no other adapter package needed, since no other adapter's workflow generation needed cycle-safe traversal logic of its own (LangGraph's `StateGraph` is itself a general graph, so no translation-layer traversal was needed there at all).
- This adapter's generated `main.py` calls `workflow.run()` synchronously (verified: Agno's `Workflow.run` has no `async` variant required for basic use, matching CrewAI's entrypoint style rather than AutoGen/Google ADK's `asyncio`-based one).
- No existing adapter, diagnostic code, or exit code changed meaning. `AGF5001` (unsupported target feature) is reused exactly as-is for this adapter's `join`/`event` blocking cases.

## Security impact

No new attack surface: this adapter follows the identical generation-time discipline every other Python adapter already has — tool and stub-step bodies are always `NotImplementedError` stubs (nothing runs until a human implements them), no credentials are ever embedded in generated code (`.env.example` documents environment variable names only), and `@agentform/compiler`'s existing `scanForSecretLeaks` gate applies to this adapter's output exactly as it does to every other target's.

## Migration impact

Additive only. `frameworkTargetSchema` gaining `'agno'` does not change validation for any specification that doesn't declare it; every existing generated project, plan file, or applied state for the original six targets is unaffected.
