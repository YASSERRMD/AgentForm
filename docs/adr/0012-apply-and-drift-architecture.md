# ADR-0012: Apply and drift architecture — atomic state mutation and a layered definition of drift

## Status

Accepted

## Context

Phase 11 (`temp/instruction.md`'s Phase 11 objectives, §15.9, §15.11) requires `agentform apply` and `agentform drift`, the first commands that actually mutate `.agentform/state.db` rather than only reading it. Three real constraints shaped the design: (1) every prior phase's read-only commands (`plan`/`status`) already established the pipeline/policy/state-backend conventions `apply` has to slot into, not replace; (2) `apply` must satisfy "cannot partially corrupt state" (Phase 11's own acceptance criterion) against a state backend (`@agentform/state-local`) that already had per-migration transactions but no general-purpose multi-write transaction primitive; (3) "drift" is used loosely across the build spec (§15.11 lists seven kinds: source, artifact, adapter-version, prompt, tool-schema, policy, runtime-deployment) without a single formula, and half of those seven have no real subsystem behind them yet (no live deployment inspection exists at all).

## Decision

- **`agentform apply` is an 11-step sequence, each step's ordering load-bearing**: acquire the state lock → load-or-verify a saved plan (if given) → revalidate the current source from scratch (a saved `.afplan` only proves "valid when made," not "valid now") → recompute the plan fresh against current deployed state → compare the fresh plan's fingerprint against the saved plan's, rejecting a stale one → re-run policy (never skipped, not even with `--auto-approve`) → confirm any `CRITICAL`-risk items interactively (or fail non-interactively without `--auto-approve`) → back up state → generate artifacts → deploy via `adapter.deploy()` where implemented → run smoke tests against `spec.evaluations.datasets` → persist every resource-state write and the new `ApplicationState` atomically → release the lock in a `finally`. A failure at any step after the backup still leaves state exactly as it was before this apply — proven by `apply.test.ts`'s smoke-test-failure case checking that persistence never ran and the lock still released.
- **Plan staleness is checked by comparing sorted `{resourceAddress, operation, risk}` fingerprints, not full `PlanItem` equality.** A saved plan and a freshly recomputed one can differ in irrelevant ways (e.g. non-deterministic map iteration order before sorting) while still describing the same decision; fingerprinting the fields that actually determine what `apply` is about to do — not the full `after` payload — is both sufficient for staleness detection and immune to incidental serialization differences.
- **`withTransaction<T>(fn)` is a new `StateBackend` method** (`@agentform/state`), mirroring `withLock`'s acquire/run/release-on-throw shape but for atomic multi-write persistence rather than mutual exclusion. `apply`'s final persistence step — every changed `ResourceState`, the new `ApplicationState`, and finishing the apply-history record — runs inside exactly one call to this, which is what makes "apply cannot partially corrupt state" true by construction rather than by discipline across several separate write calls.
- **`ApplicationState` gains a non-optional `driftStatus: DriftStatus` field (`'unknown' | 'in_sync' | 'drifted'`) plus an optional `driftCheckedAt`.** `putApplicationState` (the full-record upsert `apply` uses) always resets both to `'unknown'`/`undefined` on every call — a fresh apply invalidates any prior drift computation, since the newly-applied state _is_ the new baseline. `recordDriftStatus` is a separate, narrow mutation touching only these two fields, which is what `agentform drift` itself uses instead of a full upsert.
- **Drift is layered, not a single check**: "resource drift" is exactly `comparePlan`'s non-`NO_OP` output — the specification and deployed state disagreeing is already Terraform's own definition of drift, and `@agentform/planner` already computes it for `agentform plan`. Three more CLI-layer-only checks are added on top, each real and independently meaningful: environment drift (declared `runtime.environment` vs. what was last actually applied), adapter-version drift (the installed adapter's `manifest.version` vs. what's recorded in `ApplicationState.adapterVersions`), and artifact drift (the on-disk `generated/<target>/manifest.json`'s `irHash` vs. the current IR's `contentHash`). The other three kinds §15.11 lists — prompt drift, tool-schema drift, and runtime-deployment drift — are **not** separately implemented: prompt/tool-schema drift is already exactly what resource drift detects (a prompt or a tool schema is part of a resource's `contentHash`), and runtime-deployment drift needs a live deployment to inspect, which no adapter implements (`inspectExisting`/`deploy`/`destroy` remain unimplemented across all six). Claiming a distinct check for something `comparePlan` already covers, or fabricating a runtime-inspection check against infrastructure that doesn't exist, would both be dishonest scope-padding rather than real detection.
- **`--exit-code` is opt-in for `DRIFT_DETECTED` (12); without it, drift is reported but the exit code stays 0.** This mirrors `plan`'s own precedent (§9: "there are changes" is not itself a failure) — a routine drift-monitoring cron job wants `--exit-code` to actually alert; an interactive human running `agentform drift` to look around does not want a non-zero exit for merely looking.
- **`agentform status`'s `Drift:` line reads the _cached_ `driftStatus`/`driftCheckedAt` from `ApplicationState`, never recomputing live.** A real drift check costs as much as `agentform drift` itself (the full pipeline, policy, and — once artifact drift is added — a filesystem read); making `status` (documented as a fast, read-only overview) pay that cost on every invocation would misrepresent what kind of command it is.

## Alternatives considered

- **A single boolean "has drift" check instead of layered categories**: rejected — §15.11 explicitly enumerates multiple kinds of drift with different remediation implications (re-apply vs. re-install an adapter vs. re-run compile), and collapsing them into one boolean would lose exactly the information a human needs to know what to do next.
- **Implementing "prompt drift" and "tool-schema drift" as separate checks that re-derive them from the IR**: rejected as redundant — both are already fully covered by resource drift's `contentHash` comparison; a second check computing the same answer a different way would be duplicated logic with no new detection power, just a second place for the two to disagree if either has a bug.
- **Fabricating a "runtime-deployment drift" check that always reports "unknown" or "not applicable"**: rejected in favor of simply not listing it as a check `agentform drift` performs — a placeholder field that can never say anything else is worse than an honest gap, and `docs/security/threat-model.md`/this ADR both document why it's absent rather than papering over it.
- **Making `agentform drift`'s exit code always reflect drift status (no opt-in flag)**: rejected — would break the "plan/drift report, don't gate" precedent every prior read-only command establishes, and would make `agentform drift` unusable as a routine "let me look" command without triggering script/CI failures.
- **A full-file `restoreBackup`-style transaction for `apply`'s persistence step**: considered and rejected in favor of the narrower `withTransaction<T>` — `apply` only ever needs to atomically commit a bounded set of writes it already computed, not replace the entire database file; a full-file swap is the disaster-recovery primitive `restoreBackup` already is (see ADR-0013), a different operation with different blast radius.

## Consequences

- `StateBackend` gains `withTransaction`, `recordDriftStatus`, plus backup listing/reading methods added in the same phase for rollback (ADR-0013) — every future backend implementation (`@agentform/state-postgres`, Phase 12) must implement all of them, not just the original CRUD surface.
- `ApplicationState`'s shape changes (new required `driftStatus` field) — every existing caller that constructs one (only `apply.ts` did, until now) had to be updated; no external persisted format existed yet to migrate, since Phase 11 is the first phase anything is actually written by `apply`.
- A real, working `agentform apply` finally gives the tamper-evidence mechanisms Phase 7/10 built (`.afplan` verification, `.agentform/test-results.json`'s gate status) something that actually enforces them at blocking severity, rather than only detecting-and-warning — `docs/security/threat-model.md`'s "Plan tampering" and "Evaluation gate tampering" entries move from detection-only to enforced in this phase.
- Drift detection's honesty about what it does _not_ check (runtime-deployment drift, since no adapter can inspect a live deployment) means a user relying on `agentform drift` alone cannot detect out-of-band changes made directly against a deployed system outside of Agentform — that gap closes only once at least one adapter implements `deploy`/`inspectExisting` for real.

## Security impact

`apply`'s state-mutation path is the first place in Agentform's history any policy/plan/evaluation-gate check can actually block an operation rather than only report it — `withTransaction` ensures a blocked or failed apply leaves `state.db` exactly as it was, and the pre-mutation `createBackup()` gives every apply an undo point even for the (still theoretically possible) case where in-progress mutation is interrupted uncleanly. `docs/security/threat-model.md` is updated alongside this ADR: "Plan tampering" and "Evaluation gate tampering" move from "detection only, nothing blocks yet" to fully mitigated, and "State tampering" gains a note that `apply`'s persistence step is now transactional end-to-end, not just per-migration.

## Migration impact

`@agentform/state`'s `StateBackend` interface and `ApplicationState` type both gain new required members — `@agentform/state-local`'s `SqliteStateBackend` is the only implementation today and was updated in the same commit (task #83), so nothing is left non-compiling. `agentform apply`/`agentform drift` are new commands; no existing command's behavior or exit codes change.
