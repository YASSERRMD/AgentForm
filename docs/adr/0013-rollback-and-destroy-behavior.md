# ADR-0013: Rollback and destroy behavior ÔÇö read-only snapshots, and history that's added to but never erased

## Status

Accepted

## Context

Phase 11 (┬º15.13, ┬º15.14) requires `agentform rollback` and `agentform destroy`, alongside `apply`/`drift` (ADR-0012). Both are the second and third commands (after `apply`) that mutate `.agentform/state.db`, and both carry an acceptance criterion neither `apply` nor `drift` has: rollback "must never delete audit history," and destroy must "preserve audit history" while still requiring "explicit confirmation" and never running "silently." `@agentform/state`'s only existing full-state-restore primitive, `restoreBackup(backupId)`, replaces the entire database file ÔÇö including every `apply_history` row written since that backup, which is exactly what rollback is forbidden from doing.

## Decision

- **`readBackupSnapshot(backupId)` is a new, separate `StateBackend` method from `restoreBackup(backupId)` ÔÇö rollback uses only the former.** `readBackupSnapshot` opens the backup file in isolation (its own `openDatabase` call, closed in a `finally`), reads back its `application_state`/`resource_states` content through the exact same row-mapping functions the live database uses, and never touches the live database, its `apply_history` table, or anything else ÔÇö a pure read. `restoreBackup` remains what it always was: a disaster-recovery primitive for "state.db is corrupted or unusable, replace it wholesale," explicitly documented as not what rollback uses, since replacing the whole file would destroy every apply-history row recorded since the backup being restored to.
- **Rollback computes a diff (`toRemove`/`toRestore`) between current resource states and the snapshot's, applies just that diff inside `withTransaction` (ADR-0012), then appends a _new_ `apply_history` entry describing the rollback ÔÇö it never touches, edits, or removes any existing entry.** This is what "never erases audit history" means concretely: the row count only ever grows. `rollback.test.ts`'s dedicated test proves this directly ÔÇö two applies plus one rollback leaves exactly three history entries, all `'succeeded'`.
- **Three selectable rollback targets, resolved by a pure, separately-unit-tested function (`resolveRollbackTarget`)**: `--snapshot <backupId>` (direct, bypassing history entirely), `--to <applyId>` (looked up in `listApplyHistory()`), or the default ÔÇö the most recent history entry's `backupId`, i.e. "undo the last apply." Keeping target resolution pure and side-effect-free (never touching the live database) makes all three paths unit-testable without spinning up a real CLI process or state backend.
- **Rollback always regenerates artifacts from the _current_ on-disk specification, never from a stored historical one** ÔÇö Agentform never persists a raw historical specification anywhere (only content hashes, per ┬º10's "never store raw values"), so there is nothing else to regenerate from. When the snapshot's recorded `irHash` no longer matches the current specification's, this is surfaced honestly as `regenerationStale: true` rather than silently producing artifacts that don't actually correspond to the state just restored.
- **`agentform destroy` computes its plan independently of `comparePlan`** (`planDestroy`, `@agentform/planner`): every currently-tracked resource becomes a `DELETE` item, regardless of what the current specification declares ÔÇö or even whether a loadable specification exists at all. This mirrors `terraform destroy` tearing down everything in state, not just whatever was removed from configuration, and it means destroy has no dependency on `loadAndBuildIR` succeeding: the whole point of destroy is available even when the specification that produced the tracked state is now broken, deleted, or otherwise unusable ÔÇö plausibly the exact situation that makes someone want to destroy in the first place. `planDestroy` reimplements `classifyRisk`'s two-line `DELETE` rule (workflow ÔåÆ `CRITICAL`, else `HIGH`) rather than calling it, since `classifyRisk` requires a full `AgentformIR` destroy has no other reason to load.
- **Destroy's confirmation gate is unconditional ÔÇö every destroy with at least one tracked resource requires confirmation (interactive "yes", or `--auto-approve`), regardless of individual item risk.** This differs from `apply`, which only interrupts for `CRITICAL`-risk items specifically; ┬º15.14's "require explicit confirmation" and the Phase 11 acceptance criterion "destroy cannot run silently" are both unconditional statements, not risk-gated ones.
- **Destroy distinguishes what's recoverable from what isn't, and only warns about the latter.** Resource _state_ is backed up before every destroy (the same `createBackup()` safety net every mutating command uses) and is therefore recoverable via `agentform rollback --snapshot <backupId>` ÔÇö it is deliberately _not_ listed as something that "cannot be recovered." Generated artifact directories being deleted from disk, and (once an adapter implements `destroy`) a live deployment being torn down, genuinely have no Agentform-level undo, and are exactly what the pre-confirmation warning names.
- **Neither rollback nor destroy re-runs the full apply pipeline (smoke tests, generation-then-deploy) the way `apply` does.** Rollback regenerates artifacts (so the on-disk project matches restored state) but does not re-run smoke tests ÔÇö there is no new "desired" configuration being applied, only a return to a previously-known state, so re-validating against evaluation datasets designed for the _current_ specification would be checking the wrong thing. Destroy does not regenerate anything at all; it removes.
- **No exit code is reserved specifically for destroy failures.** ┬º14's exit-code table (`temp/instruction.md`) stops at 15 (`ROLLBACK_FAILURE`) despite discussing destroy in the same phase ÔÇö the table's own silence here is read as "destroy shares the general apply-failure bucket," so destroy failures (declined confirmation, a deployment-cleanup failure, an unexpected error) all use `APPLY_FAILURE` (10), the same code any other "a state-mutating operation failed" case uses.

## Alternatives considered

- **Using `restoreBackup` for rollback, accepting that it also discards subsequent apply-history rows**: rejected outright ÔÇö this is precisely the acceptance criterion ┬º15.13 forbids ("rollback must never delete audit history"), not a judgment call.
- **Giving destroy its own reserved exit code (e.g. 16)**: rejected ÔÇö the build spec's exit-code table is explicit and closed at 15; inventing a new code not in that table would be scope creep on a contract ┬º14 defines exactly, for a distinction (destroy-failure vs. apply-failure) the spec itself apparently didn't think warranted a separate code.
- **Requiring destroy to load and validate the current specification before proceeding, matching apply/rollback's "revalidate source" discipline**: rejected ÔÇö destroy's job is to tear down what's _tracked in state_, not what's currently _declared_; requiring a valid specification would make destroy unusable in exactly the scenario where it's most likely to be needed (a broken or already-partially-deleted project).
- **Treating resource-state backups as "cannot be recovered" alongside generated artifacts, for a simpler one-size-fits-all warning**: rejected as less honest and less useful ÔÇö state genuinely _is_ recoverable (via a fresh `agentform rollback --snapshot`), and telling the user otherwise would either cause needless alarm or train them to distrust a warning that's usually accurate for the things it does list.
- **Re-running smoke tests after a rollback**: rejected ÔÇö a dataset's assertions are written against the _current_ specification's expected behavior; running them against a workflow that was just torn back down to an older state would produce confusing, likely-failing results that say nothing meaningful about whether the rollback itself succeeded.

## Consequences

- `StateBackend` gains `readBackupSnapshot`/`listBackups` (rollback) and reuses `withTransaction`/`createBackup`/`recordApplyStart`/`recordApplyFinish` (already added for apply, ADR-0012) ÔÇö every future backend implementation must support isolated backup reads in addition to live CRUD.
- `apply_history` is now a strictly append-only audit log across three different operations (apply, rollback, destroy) that all write into the same table with different `summary` text distinguishing them ÔÇö a reader (`agentform status`'s "Last apply" line, or direct inspection) sees the most recent _any_ state-mutating operation, not specifically the most recent `apply`, which is a deliberate simplification: introducing a second history table per operation kind would fragment the audit trail Phase 11's acceptance criteria are actually about preserving.
- Destroy's independence from `loadAndBuildIR` means it has no `--target`/`--environment` override flags the way apply/rollback do ÔÇö there is no "current specification" for those flags to apply to; the targets destroy acts on come entirely from `ApplicationState.adapterVersions`'s recorded keys.
- A rollback's `regenerationStale: true` flag is informational only ÔÇö it does not block the rollback, since the state restoration itself (the part rollback is actually for) is not the part that becomes stale; only the freshly-regenerated artifacts might not perfectly represent the exact code that was live at the snapshot's time.

## Security impact

Both commands extend the same transactional/backed-up mutation discipline ADR-0012 establishes for apply: `withTransaction` for atomic resource-state writes, `createBackup()` before any destructive step. Rollback's read-only `readBackupSnapshot` is specifically what makes "never erase audit history" a structural property rather than a discipline that could regress ÔÇö there is no code path in `rollback.ts` that can reach `restoreBackup`. Destroy's unconditional confirmation gate and explicit "cannot be recovered" listing are the concrete mechanism behind "destroy cannot run silently." `docs/security/threat-model.md` is updated alongside this ADR with new entries for backup/rollback integrity and destroy irreversibility.

## Migration impact

`@agentform/state`'s `StateBackend` interface gains `readBackupSnapshot`/`listBackups`; `@agentform/state-local`'s `SqliteStateBackend` implements both in the same commit that added them (task #83), alongside `apply`'s `withTransaction`. `agentform rollback`/`agentform destroy` are new commands; no existing command's behavior or exit codes change. `apps/cli/src/lib/apply-confirm.ts` was renamed to `confirm-prompt.ts` and generalized (a `confirmAction` primitive alongside the existing `confirmCriticalChanges`) specifically to support rollback's and destroy's own confirmation prompts without duplicating the readline-based logic apply already had.
