# ADR-0010: Phase 9 adapter architecture — honest per-framework delegation, a fourth verification ecosystem, no invented uniformity

## Status

Accepted

## Context

Phase 9 (`temp/instruction.md`'s Phase 9 objectives) requires the four remaining `FrameworkAdapter`s — Microsoft Agent Framework, Google ADK, AutoGen, CrewAI — wired into `agentform compile`, completing all six §4 targets. ADR-0009 (Phase 8) established the governing discipline: block generation on any `unsupported` compatibility entry, never emit a real credential, fail every stub loudly, and verify every generated construct against the real installed target package rather than training-data recall. Phase 9 had to decide how far that discipline extends into genuinely new territory: a fourth language/ecosystem (C#/.NET, via Microsoft Agent Framework — the first non-TypeScript, non-Python target), and four frameworks whose agent-delegation models are structurally _different from each other_, not just differently-scoped subsets of the same concept the way OpenAI's "basic" and LangGraph's "broader" workflow-node support were in Phase 8.

## Decision

- **Agent delegation is reported at whatever level each framework's real, verified mechanism actually supports — never normalized to a single uniform answer across adapters.** Direct verification against each installed package found four genuinely different shapes: Microsoft's `HandoffWorkflowBuilder.WithHandoffs(...)` precisely represents a per-agent allowlist with no sharing restriction (`supported`, the most faithful of the four, but gated on a real reachability requirement — see below); Google ADK's `sub_agents` enforces a single-parent tree, so two agents naming the same delegate is a real `pydantic.ValidationError` (`unsupported` for that specific structural hazard, `supported` otherwise); AutoGen's `handoffs=` on `AssistantAgent` is real and unconstrained; CrewAI's `allow_delegation=True` is real but crew-wide — any enabled agent can reach _every_ crew member, not only its declared allowlist (`partial`, non-blocking). Reporting these differently is the honest outcome of verification, not inconsistency to be smoothed over — the cross-adapter compatibility matrix (`docs/compiler-reference.md`) states this divergence explicitly rather than picking one framework's semantics to describe all four.
- **Microsoft's handoff-source reachability requirement is detected and blocked before generation, the same structural-hazard pattern ADR-0009 didn't need but Phase 9's richer delegation models do.** Verified directly: `HandoffWorkflowBuilder.Build()` throws a real `InvalidOperationException` ("unreachable executors") unless every handoff source is reachable from the workflow's entrypoint through some chain of handoff edges. `computeHandoffReachability` in `@agentform/adapter-microsoft`'s `compatibility.ts` performs the same reachability BFS the framework does internally, reporting an unreachable source as a blocking diagnostic — mirroring `@agentform/adapter-google-adk`'s `findSharedDelegationTargets` from the same phase's own precedent one adapter earlier, and confirming (per ADR-0009's own generation-trusts-compatibility-validation boundary) that `generate-workflow.ts` doesn't need its own defensive re-check.
- **`humanApproval` is classified per-adapter based on what each framework's real mechanism actually is, not forced into a single category.** AutoGen gets `emulated`: a real `UserProxyAgent` participant genuinely exists and is repurposed to pause the conversation. ADK, CrewAI, and Microsoft all get `unsupported`, for three distinct, verified reasons rather than one shared excuse: ADK's `FunctionTool(require_confirmation=True)` and Microsoft's `ApprovalRequiredAIFunction` are real but tool-scoped, not workflow-node-scoped; CrewAI's `Task(human_input=True)` (verified against the installed package's own `SyncHumanInputProvider` source) is a review-and-refine loop scoped to one task, not a binary approval gate. In every `unsupported` case, inventing a binding from an Agentform workflow node to a specific tool call or task would be fabrication, not translation — consistent with ADR-0009's stub-vs-real-code line.
- **Model-client construction uses one of two strategies per framework, chosen by what the framework's own `model`/`llm` parameter actually is — not decided uniformly.** Where the parameter is a plain string the framework resolves itself (ADK's `model=`, CrewAI's `llm=`), the real value is always emitted; a TODO comment is added only for providers outside a small, individually-verified "resolves without extra install" set, never a stub. Where the parameter needs a real client _object_ (AutoGen's `ChatCompletionClient`, Microsoft's `IChatClient`), it's always a throwing stub, matching Phase 8's LangGraph precedent — Agentform's `model.provider` is free-form, so there is no way to derive which concrete client type a given provider needs without guessing.
- **Agents are always built via a factory function, never a module-level constant, across all four adapters — for language- and framework-specific reasons verified independently, not copied blindly from Phase 8.** AutoGen and ADK needed it to avoid a real Python import-time crash (a raising model-client stub, or — verified separately via a direct two-file reproduction — a real circular-import `ImportError` between mutually-delegating ADK agent modules). CrewAI needed it because `Agent.__init__` resolves `llm=` into a real client class _eagerly_, verified directly. Microsoft's C# static classes have no import-time side-effect hazard the way Python `import` does, but the same `Build()`-factory shape was adopted anyway for consistency and because C# has its own real (if narrower) risk in the same family: static-initializer ordering.
- **Microsoft's C#-specific codegen helpers (`json-schema-to-csharp.ts`, `csharp-identifiers.ts`) live inside `@agentform/adapter-microsoft`, not `@agentform/compiler`.** Every Python-targeting adapter shares `jsonSchemaToPythonParams`/`toIdentifier`/`pythonStringLiteral` from the compiler package precisely because four different adapters need the identical conversion; Microsoft is the only C# consumer, so hoisting its equivalents into the shared package would be a premature abstraction with a single caller.
- **No new C#-syntax-validity test exists parallel to `isSyntacticallyValidPython`/the OpenAI adapter's real-TypeScript-compiler check.** Building one would mean shelling out to a Roslyn-based syntax checker — a materially bigger lift than reusing an already-verified Python parser, and disproportionate for a single adapter. The substitute is a real, manual `dotnet build` (zero errors, zero warnings) against a fully generated project, plus running it to the documented stub, plus a separate real execution of the exact `HandoffWorkflowBuilder` shape against a fake `IChatClient` producing genuine multi-turn conversation events — the deepest verification of any single adapter in this codebase, compensating for the missing fast unit-level check with a slower but stronger end-to-end one.
- **`agentform compile`'s `ADAPTER_REGISTRY` and `ALL_FRAMEWORK_TARGETS` collapse into one list once all six targets are registered**, and the "not yet supported" error path plus the JSON output's `skippedTargets` field are removed rather than kept as permanently-unreachable defensive code — confirming ADR-0009's own prediction ("adding a Phase 9 adapter to the registry is the only change needed") rather than leaving dead branches a future reader would have to puzzle over.

## Alternatives considered

- **Reporting delegation as a single uniform level (e.g. always `partial`, to be conservative) across all four adapters**: rejected — this would understate Microsoft's genuinely precise `HandoffWorkflowBuilder` mapping and overstate CrewAI's crew-wide one identically, actively hiding the real difference a spec author needs to know before choosing a target for a multi-agent design that leans on delegation.
- **Implementing Microsoft's `humanApproval` via the low-level `RequestPort<TRequest, TResponse>` executor-graph primitive**, which is structurally the closest thing to a real node-level pause/resume mechanism of any of the six targets: rejected for this phase — it requires abandoning `AgentWorkflowBuilder`'s agent-level convenience builders for the framework's raw `WorkflowBuilder`/`Executor<TIn,TOut>` graph API, synthesizing a request/response type pair per `humanApproval` node, and verifying how the two APIs compose — a materially larger scope increase than any other adapter's hardest capability took in this phase. Left `unsupported`, documented honestly, rather than attempted partially.
- **A shared `@agentform/adapter-csharp-utils` package, or hoisting the C# helpers into `@agentform/compiler`**: rejected — one consumer does not justify a new shared package, and `@agentform/compiler` is exported to (and versioned for) every adapter; adding C#-only exports there would be dead weight for the five non-C# adapters.
- **A `csc`/Roslyn-based fast syntax checker for CI-level "every generated .cs file is syntactically valid" tests**, matching every Python/TS adapter's precedent exactly: rejected for now as disproportionate new infrastructure (a compiled helper tool, a new .NET SDK dependency in CI) for a single adapter; the real `dotnet build` verification performed manually is strictly _stronger_ per-run (catches semantic errors a syntax check wouldn't) but isn't automated into the fast unit-test loop or CI — a real, accepted gap, not a hidden one (see Consequences).

## Consequences

- The compatibility matrix in `docs/compiler-reference.md` is genuinely uneven for agent delegation across the six targets — a spec author choosing between Microsoft and CrewAI for a delegation-heavy design gets an accurate, differentiated answer instead of a smoothed-over one.
- `@agentform/adapter-microsoft`'s and `@agentform/adapter-google-adk`'s `compatibility.ts` both carry a real structural-hazard detector beyond a flat node-type map (`computeHandoffReachability`, `findSharedDelegationTargets`) — any future adapter whose native delegation/handoff mechanism has its own real constraint should follow the same pattern: detect it structurally in `compatibility.ts`, block it as `unsupported`, and let `generate()` trust that gate rather than re-checking.
- Microsoft-generated projects have no automated CI syntax/build check today — a change to `@agentform/adapter-microsoft`'s codegen could regress without CI catching it until someone runs `dotnet build` by hand. This is an explicit, accepted gap for this phase (see Alternatives), not an oversight; a future phase could close it by adding a .NET SDK setup step to `.github/workflows/ci.yml` if the tradeoff (CI runtime, a new toolchain dependency) is judged worth it.
- `agentform compile --json`'s output shape drops the `skippedTargets` field that Phase 8 shipped (always `[]` from the moment all six adapters were registered) — a real, if inconsequential, output-shape change; noted here since the repository has no version tag/release yet for this to be a breaking change against.
- Every one of the four new adapters' real-installation verification surfaced at least one genuine gotcha before it shipped (AutoGen's bare-model-client-string trap, ADK's circular-import and single-parent-tree hazards, CrewAI's eager-`llm`-resolution and bare-string-defaults-to-OpenAI traps, Microsoft's handoff-reachability requirement) — continuing to demonstrate, four adapters in a row, that ADR-0009's "verify against the real package" discipline finds real bugs a plausible-looking implementation would not have caught.

## Security impact

No new attack surface. All four new adapters continue every mitigation ADR-0009 established: `scanForSecretLeaks` runs over their generated files exactly as it does OpenAI's/LangGraph's (proven by `apps/cli/src/cross-target.test.ts` compiling all six for real), no adapter ever emits a real credential value (each documents required environment variables in `.env.example` instead), and every stub fails loudly (`raise NotImplementedError`/`throw new NotImplementedException`) rather than silently doing nothing. See `docs/security/threat-model.md`, updated alongside this ADR.

## Migration impact

None — `@agentform/adapter-microsoft`, `@agentform/adapter-google-adk`, `@agentform/adapter-autogen`, and `@agentform/adapter-crewai` are new packages. `apps/cli/src/commands/compile.ts`'s `ADAPTER_REGISTRY` is extended and its now-unreachable "not yet supported" validation path removed; the `agentform compile --json` output's `skippedTargets` field is removed (see Consequences). No existing package's public shape changes.
