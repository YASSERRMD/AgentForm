# ADR-0009: Compiler and adapter architecture ‚Äî blocking compatibility, honest stubs, verified-not-assumed generation

## Status

Accepted

## Context

Phase 8 (`temp/instruction.md` ¬ß12, ¬ß13.1, ¬ß13.2, and Phase 8's own objectives) requires a compiler architecture, a generated-project abstraction, deterministic file generation, compatibility reports, a manifest matching ¬ß22's exact shape, and two framework adapters ‚Äî OpenAI Agents SDK and LangGraph ‚Äî plus `agentform compile`. The spec gives the `FrameworkAdapter` interface shape and the manifest JSON example, but leaves open: how strictly to enforce "do not silently ignore unsupported specification fields" (¬ß12), how much of each adapter's generated code should be real versus a scaffold, how to derive an agent's model-calling logic when Agentform's `model.provider` is a free-form string, and how to actually know generated code works rather than merely looking plausible. Two adapters targeting genuinely different kinds of SDKs (OpenAI's implicit, no-explicit-graph model versus LangGraph's explicit state-graph model) also meant the two adapters ended up with different internal shapes ‚Äî this ADR records why that's intentional, not drift.

## Decision

- **A blocking incompatibility stops generation entirely ‚Äî `compile()` never calls `adapter.generate()` when `CompatibilityReport.hasBlockingIncompatibility` is true.** Every `unsupported` compatibility entry becomes an `AGF5001` error diagnostic; `partial`/`emulated` entries become warnings. This is the literal, strict reading of ¬ß12's "do not silently ignore unsupported specification fields" ‚Äî a project using a node type an adapter can't generate fails loudly (`agentform compile` exit 13) rather than producing a project quietly missing part of what was declared.
- **`generatedAt` is always `null` in every manifest, never a real timestamp.** ¬ß22 is explicit that reproducibility requires this; a timestamp belongs in apply metadata (a later phase), not the deterministic artifact itself. `buildManifest` (`@agentform/compiler`) is the one place this is constructed, so every adapter gets it right by construction.
- **`manifest.json` is written by the CLI (`agentform compile`), not by either adapter.** It's metadata _about_ the generation, not part of the generated application ‚Äî neither ¬ß13.1's nor ¬ß13.2's documented file tree includes it. Writing it once, centrally, avoids duplicating the same `JSON.stringify` call (and getting it subtly wrong) in every adapter.
- **The secret-leak scan is a structural safety net, not a substitute for adapters being careful.** `${env.*}` references are fully resolved into the IR before the compiler ever sees it (Phase 3's interpolation precedes schema validation) ‚Äî the compiler has no way to trace which IR field values originated from an environment variable. Both adapters are written to never emit an IR field's literal value into a credential-shaped position, relying on the target SDK/runtime's own env-var conventions (`OPENAI_API_KEY`, etc.) instead ‚Äî but `scanForSecretLeaks` (reusing `@agentform/policy`'s `detectSecret`/`SECRET_PATTERNS`) still runs over every generated file as the final gate, and blocks `compile()` from ever returning a `project` that would contain a secret-shaped value regardless of how it got there.
- **Every stubbed function body fails loudly** (`throw new Error(...)` / `raise NotImplementedError(...)`), never silently returning a plausible-but-fake value or doing nothing. Agentform generates a project's interface, never its business logic ‚Äî a project that hasn't been filled in should fail obviously the moment it's run, not subtly.
- **OpenAI adapter: handoffs come from `agent.delegation.allowedAgents`, not inferred from workflow-node edges.** The OpenAI Agents SDK's own handoff model is agent-level config, not graph-level ‚Äî deriving it from the agent's own declared field is simpler, avoids circular-ES-module-import risk between agent files, and matches "basic multi-agent workflow" (Phase 8's own OpenAI scope) rather than attempting full workflow-graph translation the SDK doesn't need.
- **LangGraph adapter: the workflow graph (nodes + edges) is translated directly, node-for-node and edge-for-edge**, because that's what `StateGraph` actually needs ‚Äî unlike OpenAI, there is no higher-level primitive that makes explicit graph wiring unnecessary. Edge _topology_ decides `add_edge` vs. `add_conditional_edges` (one unconditional outgoing edge vs. anything else), independent of node type ‚Äî the same mechanism handles a `router` node's branches and a `loop` node's cycle-back edge without special-casing either.
- **LangGraph agent nodes are honest `NotImplementedError` stubs, not `langgraph.prebuilt.create_react_agent`.** `create_react_agent` would be the closest analogue to OpenAI's fully-realized `new Agent({...})`, but its `model` parameter needs either a real chat-model instance or a `"provider:model"` string that requires the full `langchain` meta-package installed (confirmed with a real `ImportError` when it isn't). Agentform's `model.provider` is a free-form string (`@agentform/schema`), so mapping it to a specific LangChain integration package per provider would mean guessing ‚Äî exactly the kind of unverified logic this codebase's adapters are built to avoid.
- **LangGraph loop nodes get a real, non-stub iteration-counter increment** (`state.py` declares one `int` field per loop node; the loop node's body increments it). This is fully mechanical ‚Äî no business logic to fabricate ‚Äî and it's what makes "Loop limit" (a required LangGraph feature) a genuine, working safety mechanism rather than only a comment, even though the loop's _continuation decision_ is still a stub (Agentform has no expression evaluator for the `when` condition driving it).
- **Every generated construct in both adapters was verified against the real installed target package, not training-data recall** ‚Äî `.d.ts` inspection and a real `npm install && npm run build` for OpenAI (`@openai/agents@0.13.5`); `inspect.signature` against real classes/functions and a real `pip install` + `python -m src.main` execution for LangGraph (`langgraph==0.6.11`, in a real virtualenv). This caught three real bugs before they shipped (see Consequences).

## Alternatives considered

- **Warn instead of block on unsupported features**: rejected ‚Äî ¬ß12's wording ("do not silently ignore") and Phase 8's own minimum test ("Unsupported feature failure") both read as requiring a hard stop, not a warning a user could miss.
- **`create_react_agent` for LangGraph agent nodes, accepting `langchain` as a generated-project dependency**: rejected ‚Äî see Decision above. Pinning a `langchain-<provider>` package per agent would require either hardcoding a provider‚Üípackage table (unverifiable for providers not checked against a real install) or asking the user to fill in a dependency Agentform itself couldn't determine, which is worse than an honest stub.
- **Evaluating workflow edge `when` conditions via `eval()`/`Function()`** so conditional-routing code could be fully real: rejected outright ‚Äî the workflow schema's own doc comment is explicit that safe expression evaluation without `eval` is out of scope until a real expression-language evaluator exists (not yet built in any phase through Phase 8); generating code that calls `eval()` on specification-supplied strings would be a direct security regression.
- **Deriving LangGraph handoffs the same way as OpenAI (`agent.delegation.allowedAgents`)**: rejected ‚Äî the workflow graph is already the authoritative source of control flow for LangGraph (StateGraph requires it explicitly), so re-deriving equivalent structure from a second, potentially-inconsistent source (agent-level delegation) would risk the two disagreeing; translating the declared graph directly is both simpler and more faithful.
- **Each adapter writing its own `manifest.json`**: rejected in favor of the CLI writing one canonical manifest per compiled target, avoiding the shape drifting between adapters over time.

## Consequences

- A project cannot be compiled for a target unless every node/tool type it uses has a real generator for that target ‚Äî adding support for a new node type (e.g. `subworkflow`) to either adapter is the only way to unblock a project that uses it, there is no escape hatch.
- Real-installation verification found and fixed three concrete bugs during this phase, all now covered by regression tests: OpenAI's `defineInputGuardrail` isn't part of the SDK's public exports (fixed by generating a plain `InputGuardrail`-typed object instead); `z.record(...)` doesn't satisfy the SDK's `ZodObjectLike` constraint for `tool()`/`Agent.outputType` (fixed by always falling back to `z.object({})`); LangGraph's `graph.invoke()` with a checkpointer attached requires a `thread_id` in the call config or raises immediately (fixed by generating a fresh `uuid4()` thread id in `main.py`). This is a demonstration, not a one-time fact ‚Äî the same "verify against the real package" discipline should apply to Phase 9's four remaining adapters.
- Any future phase adding a workflow-edge expression evaluator (safely, without `eval`) unblocks _both_ adapters' routing-function stubs simultaneously, since both were written against the same "no evaluator exists yet" constraint.
- `agentform compile`'s `--all` flag compiles only the adapters actually registered (openai, langgraph) ‚Äî adding a Phase 9 adapter to the registry is the only change needed for `--all` to pick it up; nothing else in the CLI hardcodes the list of two.

## Security impact

Continues the "never embed credentials in generated code" line from ¬ß22 directly ‚Äî both adapters rely on target-SDK env-var conventions rather than emitting values, and the secret-leak scan is an unconditional final gate regardless of which adapter produced the files (see Decision above). No adapter executes generated or specification-supplied code as part of compilation ‚Äî code generation is string assembly against verified real APIs, not templating through an interpreter. See `docs/security/threat-model.md` for the full cross-package picture (due its own Phase 8 update alongside this document).

## Migration impact

None ‚Äî `@agentform/compiler`, `@agentform/plugin-sdk`, `@agentform/adapter-openai`, and `@agentform/adapter-langgraph` are new packages, and `agentform compile` is a new command. `apps/cli`'s existing commands and `@agentform/ir`'s public shape are unchanged.
