# ADR-0008: State and planner architecture — hash-only state, node:sqlite, file locking, honest risk precision

## Status

Accepted

## Context

Phase 7 (`temp/instruction.md` §9, §10, and Phase 7's own objectives) requires a local SQLite state backend, migrations, locking, desired/current comparison, plan risk classification, saved/verifiable plan files, backups, and interrupted-apply recovery — plus explicit acceptance criteria: plan order respects dependencies, state writes are atomic, saved plans are tamper-evident, and failed planning never mutates state. The spec gives the `PlanItem`/`PlanOperation`/`PlanRisk` shapes and an illustrative risk table (prompt change: medium, model version change: high, removal of human approval: critical, etc.), but leaves open: which SQLite driver to use, how file locking actually works, and — the decision with the widest downstream effect — what a stored `ResourceState` actually contains, given §10's "never store raw secret values; store secret reference metadata only." That last constraint turns out to directly limit how precisely plan diffs and risk can be classified, which this ADR works through explicitly rather than leaving implicit.

## Decision

- **`ResourceState` stores content hashes, never resource values — not even redacted ones.** A resource's actual field values (prompt text, tool config, model settings) are never persisted to state, only a `contentHash` (the whole resource) and an `identityHash` (a small fingerprint of type/provider-style identity fields only). This is the strictest reading of "never store raw secret values" available: rather than trying to redact-then-store (which requires trusting a redaction pass to catch every secret shape, forever, across every resource kind), nothing that could contain a secret is stored at all.
- **This constraint has a real, accepted cost: `PlanItem.changes` (field-level diffs) and precise field-triggered risk rules (prompt text changed, model version bumped, network access expanded, cost ceiling raised, data residency changed) are not achievable from stored state alone**, because computing them needs the resource's *previous* actual value, which by the decision above doesn't exist anywhere. `@agentform/planner`'s risk classifier (`risk.ts`) is explicit about this in its own doc comment: it implements the *precise* §9 rules it can compute from the desired value alone (a newly created tool's `sideEffect`, a workflow with an ungated destructive-tool call — structurally identical to `@agentform/policy`'s `AF004`, adapted to the IR), and falls back to conservative operation-type baselines (`UPDATE` → `MEDIUM`, `REPLACE`/`DELETE` → `HIGH`, deleting a `workflow` → `CRITICAL`) everywhere a precise rule would need history. This is a deliberate, documented approximation, not a silent gap — see `packages/planner/src/risk.ts`'s doc comment and `docs/planner-reference.md`'s Scope section.
- **`identityHash` is a third, narrow exception to "hash-only": it exists specifically to distinguish `REPLACE` from `UPDATE`** without needing previous values — a small fingerprint (e.g. `{ type }` for a tool, `{ provider }` for a model) hashed separately from the full resource. This required revising `ResourceState`'s shape mid-phase (see the two `feat(state)` commits) once the risk/replace-detection design became concrete — an example of a type getting one iteration of real feedback before the phase's commands were built on top of it, not a design mistake carried through the whole phase.
- **`node:sqlite`'s `DatabaseSync`, not a native dependency (`better-sqlite3`) or a WASM one (`sql.js`).** `node:sqlite` is still `ExperimentalWarning`-flagged as of the Node versions this package targets, but the alternative — a native-compiled addon — means every install of this CLI depends on a working node-gyp toolchain or a prebuilt binary matching the user's exact platform/arch/Node ABI, which is a materially worse practical problem for a globally-installed CLI than one experimental-API risk contained behind `@agentform/state`'s interface. The experimental warning itself is suppressed surgically (`sqlite-module.ts`: `process.emitWarning` is intercepted by name and message content for the duration of the `node:sqlite` import only, then restored) — every other Node warning still reaches the user unfiltered.
- **File-based locking (`.agentform/lock`, exclusive create), not SQLite's own locking.** §10's example state directory lists `lock` as a file sibling to `state.db`, and file locking is genuinely simpler to reason about and test than relying on SQLite's internal (and driver-specific) locking semantics for a *cross-process* "is an apply already running" signal. Acquiring is atomic at the filesystem level (`writeFileSync` with the `wx` flag fails `EEXIST` if the file exists — no separate coordination primitive needed); a lock older than `staleTimeoutMs` (default 10 minutes) is treated as abandoned and taken over. The stale-takeover path has one narrow, accepted TOCTOU race (another process could win the recovery window) — acceptable for a local, single-machine backend, and it fails closed (`StateLockError`) rather than looping if it happens.
- **`agentform plan` never acquires the lock.** Locking exists to serialize *mutating* operations; `plan` only reads state (`listResourceStates()`), so acceptance criterion "failed planning never mutates state" holds structurally — there is no code path in `plan` that calls any `put*`/`delete*` method on the backend at all, not just a lock discipline. Locking is exercised by `@agentform/state-local`'s own test suite (including contention across two separate backend instances) rather than by the CLI; `apply` (Phase 11) is what will actually need it.
- **Plan file tamper-evidence reuses `@agentform/ir`'s `computeContentHash`/canonicalization**, rather than a second hashing implementation — a `.afplan` file's `contentHash` covers `formatVersion`/`createdAt`/`items`, computed and verified with the exact same canonical-JSON-then-sha256 approach the IR's own content hash already uses.
- **`agentform status` reports `driftStatus`/`evaluationStatus` as honest "unknown, not implemented" strings**, not fabricated data — no drift detection (`agentform drift`, Phase 7's own §15.11 is a *separate*, not-yet-built command) or evaluation engine (Phase 10) exists yet to produce real values for these fields. `policyStatus` is the one status field with a real answer today, since `@agentform/policy` already exists — reusing the exact same `evaluatePolicies` call `validate`/`plan` already make.

## Alternatives considered

- **Store redacted/sanitized resource values in state** (to enable real field-level diffing and precise risk rules): rejected — trusting a redaction pass to be exhaustive forever, across every current and future resource shape, is a weaker guarantee than "the data was never written," and §10's own wording ("never store raw secret values") reads as the stronger guarantee being the intended one.
- **Per-field hashes for every risk-relevant field** (a `riskFingerprint` hash per field the §9 table names, compared field-by-field): considered, would give a few more precise rules (e.g. real "model version changed" detection) without storing raw values. Deferred rather than rejected outright — it's a legitimate, boundedly-scoped future enhancement to `ResourceState`, not implemented now because `identityHash` already covers the one rule (`REPLACE` detection) that was load-bearing for this phase's minimum tests, and adding a second, broader hash-fingerprint mechanism for every other rule was assessed as scope beyond what Phase 7's actual acceptance criteria require.
- **`better-sqlite3`**: rejected — see Decision above (native-compilation distribution risk outweighs one contained experimental-API risk).
- **SQLite's own file locking / `BEGIN EXCLUSIVE` for cross-process coordination**: rejected — conflates "is a transaction in progress" (SQLite's job, already handled) with "is another `agentform` process running an apply" (a higher-level, longer-lived concern this needs its own explicit lock file for, matching §10's example layout directly).
- **`agentform plan` acquiring a lock anyway, "to be safe"**: rejected — locking something that never mutates state adds contention (two people running `plan` concurrently would fail for no real reason) without protecting anything.

## Consequences

- Any future phase that wants real field-level plan diffs or the remaining precise risk rules (prompt/version/network/cost/residency changes) needs to either add a bounded, explicitly-designed snapshot mechanism (the deferred per-field-hash approach above, or something else) — it is not a small tweak to today's `comparePlan`, since the underlying data genuinely isn't available yet.
- `identityHash`'s per-kind fingerprint function (`identityFingerprint` in `packages/planner/src/desired-resources.ts`) only covers `tool.type` and `model.provider` today; a future resource kind or a future "this field changing should force replacement" rule needs an explicit addition there, not an automatic one.
- `node:sqlite` remaining experimental means a future Node upgrade could change its API in a way `@agentform/state-local` needs to adapt to — contained to that one package behind `@agentform/state`'s interface, per the original design goal of keeping the interface backend-agnostic.
- `agentform status`'s `driftStatus`/`evaluationStatus` fields are expected to change from placeholder strings to real computed values once Phase 7's own drift command and Phase 10's evaluation engine exist — a future phase updating this file to replace those specific two lines is expected, not a sign of a bug today.

## Security impact

Continues the "never store raw secret values" line directly from §10 — see Decision above for why hash-only (not redact-then-store) was chosen as the stronger guarantee. File locking prevents concurrent mutating operations from corrupting state (relevant once `apply` exists in Phase 11). Plan files are tamper-evident via content hashing, verified in `packages/planner/src/plan-file.test.ts` against both a straightforward content edit and an edited-hash-field attempt. See `docs/security/threat-model.md` for how this fits the full cross-package picture (that document is due its own Phase 7 update alongside this ADR).

## Migration impact

None — this is the initial state/planner implementation. `apps/cli`'s `LoadAndBuildResult`/`BuildIRResult` (already additive from Phase 6) are unchanged; `plan`/`status` are net-new commands.
