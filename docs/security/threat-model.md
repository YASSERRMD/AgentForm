# Agentform threat model

## Purpose

Agentform is a control plane for agentic AI systems: a specification language, a compiler, and a policy engine that sit between a human author and whatever framework/runtime actually executes an agent. That position gives it two distinct attack surfaces — the _documents_ it parses and validates (which may be authored by someone untrusted, or composed from imported projects) and the _systems it will eventually drive_ (frameworks, tools, deployment targets, state). This document tracks both, following §19 of the build specification.

Status labels used below:

- **Mitigated** — a concrete, tested control exists today.
- **Partially mitigated** — a real control exists but doesn't cover the full threat (usually because the runtime half of Agentform doesn't exist yet).
- **Planned** — no control exists yet because the subsystem it would protect doesn't exist yet; a placeholder is registered where one makes sense so it can't be silently disabled later.

As of Phase 11, Agentform validates, compiles to an IR, evaluates policy, can plan and now actually **apply** (persist deployed state atomically, generate artifacts, run smoke tests), detect **drift** between specification/state/artifacts, **roll back** to a previous apply without ever erasing audit history, **destroy** tracked resources, and make a limited, best-effort attempt to **import** an existing raw project into a candidate specification. It can compile a specification into a real project for any of its six target frameworks (OpenAI Agents SDK, LangGraph, Microsoft Agent Framework, Google ADK, AutoGen, CrewAI), and can run a specification's evaluation datasets against a deterministic, fully offline mock execution engine. It still does not yet execute agents against a real model/tool provider, or deploy to/tear down/inspect a _live_ target — no `FrameworkAdapter` implements `deploy`/`destroy` yet, and `inspectExisting` (two adapters, Phase 11) only recognizes _raw source code_ for `agentform import`, never a running deployment. Several threats below are structurally about that missing runtime half; those are marked Planned rather than overclaimed as mitigated.

## Threats and mitigations

### Malicious Agentform files

A hostile YAML/JSON document crafted to exploit the parser, the schema validator, or a downstream reader. **Mitigated**: safe YAML parsing (see below), full Zod schema validation rejects anything off-shape, a maximum source file size (`AGF1010`), a maximum `$ref` chain depth (`AGF1004`), reference cycle detection (`AGF1003`), and workflow node/edge/expression-length caps (`AGF3016`–`AGF3018`) all apply before or during parsing — see [`packages/parser`](../../packages/parser) and [`packages/ir/src/semantic/limits.ts`](../../packages/ir/src/semantic/limits.ts).

### Prompt injection

Text inside an agent's instructions, a tool's description, or interpolated content that's designed to manipulate the model at run time. **Planned** (structural mitigations only): Agentform validates and compiles specifications; it does not run a model against a prompt itself, so it cannot detect injected intent at parse time — that's inherently a runtime/model-level concern, not a static-analysis one. What today's policy engine _does_ do is bound the blast radius of a successful injection: [`AF004`](../../packages/policy/src/policies/af004-critical-actions-require-human-approval.ts) requires human approval before a destructive tool runs regardless of what convinced the agent to call it, and [`AF003`](../../packages/policy/src/policies/af003-write-tools-require-explicit-permission.ts)/[`AF002`](../../packages/policy/src/policies/af002-no-unrestricted-shell-tools.ts) keep an agent's tool surface narrow. Phase 8's OpenAI adapter now generates a real, named stub per declared guardrail (`InputGuardrail`-typed, wired into the agent's `inputGuardrails`) — the _scaffolding_ for a runtime defense exists and is structurally correct, but the stub body always reports `tripwireTriggered: false` until a human fills in real detection logic, so this remains Planned for actual mitigation, not Mitigated by the stub's existence alone.

### Tool injection

A malicious or spoofed tool definition entering the resolved document, e.g. via a compromised imported project. **Mitigated** for the injection vector, **partially mitigated** for impact: every reference (`$ref`/`file`/`schemaRef`) resolves to a local path inside the project root only — there is no URL-fetching reference type, so a tool definition can't be pulled from an untrusted network source at parse time. An injected tool is still subject to the full policy pack: `AF002`/`AF003`/`AF012` limit what it can declare itself capable of.

### Compromised plugins

`customPlugin` exists as a tool _type_ in the schema (§6.4), but there is no plugin loading or execution mechanism yet — `@agentform/plugin-sdk` has no runtime implementation as of Phase 6. **Planned**: §19's "plugin trust policy" is meaningful once plugins can actually be loaded and run; until then there is nothing to compromise.

### Secret leakage

A real credential appearing in a document, a diagnostic message, a log line, a test snapshot, generated framework code, or — new in Phase 10 — an evaluation test report. **Mitigated**: [`AF001`](../../packages/policy/src/policies/af001-no-inline-secrets.ts) walks the entire document looking for known credential shapes; [`redactSecretValue`](../../packages/policy/src/redact.ts) ensures any message that names a detected secret never echoes the raw value; [`AF010`](../../packages/policy/src/policies/af010-prompt-recording-disabled-for-restricted-data.ts) blocks `recordPrompts: true` alongside restricted-classification data, and `recordPrompts` itself defaults to unset/off (§18). `@agentform/compiler`'s `scanForSecretLeaks` (reusing the same `detectSecret`/`SECRET_PATTERNS` `AF001` uses) runs over every file any of the six adapters generates and blocks `compile()` from returning a project containing a secret-shaped value, regardless of whether the value reached the IR through `${env.*}` interpolation (already fully resolved before the compiler ever sees it) or some other path — see `docs/compiler-reference.md`. `agentform test`'s evaluation datasets are a distinct vector `AF001` doesn't cover at all (they're loaded separately from the specification document `evaluatePolicies` walks): a mocked tool call's `args`/`return` value is author-controlled test-fixture content, commonly captured from a real API response while writing the mock, and would otherwise flow verbatim into a failing assertion's message and from there into console/`--json`/`--junit` output designed to feed CI dashboards that archive what they're given. `redactSecretsFromReport` (`apps/cli/src/lib/report-redaction.ts`) runs the same `SECRET_PATTERNS`/`redactSecretValue` over every fully rendered report before it's written anywhere, closing that gap — see `docs/evaluation-reference.md`.

### State tampering

Unauthorized modification of Agentform's persisted current-state record. **Partially mitigated**: [`@agentform/state-local`](../../packages/state-local) stores every mutation in a real transactional database (SQLite, atomic per-migration and, as of Phase 11, atomic per-apply/per-rollback via the new `withTransaction` — `agentform apply`'s/`agentform rollback`'s entire persistence step commits or fully rolls back together, never partially, per ADR-0012/ADR-0013), and file-based locking (`.agentform/lock`) prevents two `agentform` processes from writing concurrently — now actually held for the duration of every mutating command (`apply`/`rollback`/`destroy`), not just modeled. Every mutation is preceded by `createBackup()`, giving tampered-or-bad state a real recovery path via `agentform rollback --snapshot`. What's not yet covered: `state.db` itself has no integrity check (a hash over its contents, checked on open) and no encryption at rest (§6.6 lists encryption hooks as future work) — protection today is "hard to corrupt via normal concurrent use, and recoverable via backup if it is," not "detects a file edited directly on disk." [`AF014`](../../packages/policy/src/policies/af014-state-must-not-contain-secrets.ts) remains a mandatory placeholder for the same reason as before: nothing raw is stored in state to leak in the first place (see `docs/state-reference.md`), so there's currently nothing for a real `AF014` check to find.

### Plan tampering

Unauthorized modification of a generated plan file between `plan` and `apply`. **Mitigated**, now end-to-end: [`@agentform/planner`](../../packages/planner)'s `.afplan` files are tamper-evident — `createPlanFile`/`verifyPlanFile` compute and check a content hash (the same canonicalization [`computeContentHash`](../../packages/ir/src/hash.ts) uses) over the plan's full contents; any edit to the file, including an attempt to also edit the recorded hash to match, is caught on verify (`packages/planner/src/plan-file.test.ts` proves both cases). As of Phase 11, `agentform apply plan.afplan` actually calls `verifyPlanFile` before doing anything else with a saved plan, rejecting a tampered file with `APPLY_FAILURE` — the detection mechanism this section previously described as "tested standalone, but nothing consumes it end-to-end" now has a real, tested consumer (`apply.test.ts`'s "rejects a tampered saved plan file" case). Apply additionally recomputes the plan fresh and compares fingerprints even for a _valid, untampered_ saved plan, rejecting anything stale — a second, independent layer beyond tamper detection alone.

### Evaluation gate tampering

Unauthorized modification of `.agentform/test-results.json` to make it look like `agentform test` passed when it didn't, or ran more recently than it did. **Mitigated**, both for detection and (indirectly) for impact: [`@agentform/evaluator`](../../packages/evaluator)'s `TestResultsRecord` is tamper-evident the same way a `.afplan` file is — `createTestResultsRecord`/`parseTestResultsRecord` compute and check a content hash over the record's full contents, and the record's own `irHash` lets a reader tell "these results are for the specification as it exists right now" apart from "these results are for some earlier version of it" (`checkEvaluationGateStatus`, `docs/evaluation-reference.md`). `agentform plan`/`agentform status` both read this back and treat an invalid or tampered file exactly like no file at all — never trusting unverifiable content. `agentform plan`'s own diagnostics (`AGF6001`–`AGF6003`) remain `warning` severity by design (ADR-0011) — `plan` is a dry-run, and nothing it reports blocks anything. What changed in Phase 11: `agentform apply` doesn't read the old results file's tamper-evidence at all — it re-runs the actual smoke tests fresh, every time, against the current specification, and a failure genuinely blocks the apply (exit 9), persisting nothing. This sidesteps "an attacker tampers with the results file to fake a pass" as a concern for `apply` specifically (there's nothing to tamper with that `apply` would trust), while `plan`/`status`'s advisory reporting continues to rely on the tamper-evident record as before.

### Rollback/backup integrity

Unauthorized or accidental use of `agentform rollback` to silently discard legitimate deployed state, or a rollback that itself corrupts state or erases audit history. **Mitigated**: `agentform rollback` never uses `restoreBackup` (the full-file-replace disaster-recovery primitive, which _would_ discard every `apply_history` row written since the backup) — it uses `readBackupSnapshot`, an isolated read that never touches the live database, computes a diff, applies it inside `withTransaction`, and appends a _new_ history record describing the rollback (`rollback.test.ts`'s "never erases audit history" test proves three operations produce exactly three preserved history rows, not fewer). A rollback itself is preceded by `createBackup()`, so a bad rollback has its own undo point. Confirmation is required the same way `apply`'s critical changes are (interactive "yes" or `--auto-approve`; a non-interactive, non-approved rollback changes nothing and exits 15) — a rollback cannot happen silently in an automated context without explicit opt-in. See ADR-0013.

### Unintended or silent destruction

`agentform destroy` removing tracked resources, generated artifacts, or (once implemented) a live deployment without the operator clearly understanding the blast radius. **Mitigated** for the parts of the blast radius Agentform controls today: confirmation is unconditional — _any_ destroy with at least one tracked resource requires an interactive "yes" or `--auto-approve`, never risk-gated the way `apply`'s confirmation is, and a non-interactive, non-approved destroy changes nothing (proven: `destroy.test.ts`'s declined-destroy test checks resource count and the on-disk `generated/` directory are both unchanged afterward). The pre-confirmation message explicitly separates what's recoverable (tracked resource state, backed up first, restorable via `agentform rollback --snapshot`) from what isn't (generated artifact directories about to be deleted from disk; a live deployment, once an adapter implements `adapter.destroy()`) — an operator sees the honest, real blast radius before confirming, not a generic warning. **Planned** for the deployment half: since no adapter implements `destroy` yet, there is no live infrastructure Agentform can actually tear down (or fail to warn about) today — the hook is wired and this section will need re-evaluating for real consequences once one exists.

### Generated-code tampering

Unauthorized modification of framework code Agentform generates. **Partially mitigated**: generation is deterministic — the same IR always produces byte-identical files (proven for every adapter: each one's `adapter.test.ts` has a "two `generate()` calls produce byte-identical files" test), and every generated project carries a manifest recording the exact `sourceHash`/`irHash` that produced it (§22, `generatedAt` always `null`). Together these make tampering _detectable_ by recompiling from the same source and diffing against the manifest's recorded hashes — but nothing automated does this today; there is no `agentform compile --verify`-style command, and the manifest hashes the _inputs_ to generation, not the generated files themselves. [`AF015`](../../packages/policy/src/policies/af015-generated-code-must-be-reproducible.ts) (registered since Phase 6) is exactly the policy this phase gives real teeth to — a non-deterministic adapter would now actually fail its own test suite, not just violate an aspirational policy.

### Unsupported specification fields silently dropped

A compiler that generates a project missing part of what the specification actually declared, without saying so — the generated system would then diverge from the declared desired state in a way nothing surfaces. **Mitigated**: every adapter's `validateCompatibility` reports every workflow node type and tool type against what it can actually generate; any `unsupported` entry sets `hasBlockingIncompatibility`, and `compile()` refuses to call `generate()` at all when that's set — returning diagnostics only, never a partial project (§12 "do not silently ignore unsupported specification fields," enforced structurally, not by adapter discipline alone). `agentform compile` surfaces this as exit code 13 with a diagnostic naming the specific unsupported feature. See `docs/compiler-reference.md`.

### Dependency confusion

A malicious package with a name similar to a real Agentform package or plugin getting installed instead. **Mitigated** at the tooling level, not an Agentform-product level: every package is scoped (`@agentform/*`), the workspace lockfile is committed, and CI installs with `pnpm install --frozen-lockfile` ([`.github/workflows/ci.yml`](../../.github/workflows/ci.yml)) rather than resolving fresh ranges.

### Path traversal

A reference that tries to resolve outside the project root. **Mitigated**: every file reference — `$ref`, `file`, `schemaRef`, the entry file, environment overlays, auto-discovered resources — goes through [`resolvePathWithinRoot`/`resolvePathRelativeToFile`](../../packages/core/src/safe-path.ts), which reject absolute paths and any `..` chain that escapes `rootDir` as a diagnostic (`AGF1002`), never a thrown exception. Proven against a real on-disk fixture in [`specifications/v1alpha1/security/malicious-path`](../../specifications/v1alpha1/security/malicious-path).

### Unsafe file references

The general case of the above: any reference mechanism that could read something it shouldn't. **Mitigated** by the same sandbox — `file`/`schemaRef`/`$ref` are the _only_ reference mechanisms, all local-path-only, all sandboxed identically.

### Arbitrary code execution

Something in a document causing Agentform's own process to execute attacker-controlled code. **Mitigated** for Agentform itself: no `eval`, `Function()`, or dynamic `import()` anywhere in this codebase (see ADR-0004); YAML parsing never executes custom tags. **Out of scope by design**: whether a _generated_ `function`/`customPlugin` tool itself wraps shell execution is an application-author decision Agentform can flag ([`AF002`](../../packages/policy/src/policies/af002-no-unrestricted-shell-tools.ts)) but not prevent outright — and there is no Agentform runtime executing tools yet regardless.

### SSRF through HTTP tools

An `http`/`openapi` tool's destination pointing at an internal service or a cloud metadata endpoint (e.g. `169.254.169.254`). **Partially mitigated**: [`AF012`](../../packages/policy/src/policies/af012-network-destinations-must-be-allowlisted.ts) requires `networkDestination` to be explicitly declared for network-capable tools, closing the "unconstrained because undeclared" gap. There is no request-time enforcement yet (blocking specific destinations at the moment a call is made) because there is no HTTP-tool runtime yet — that lands with the adapter/runtime work.

### Excessive network permissions

A tool or agent granted broader network reach than it needs. **Partially mitigated** by the same `AF012`, plus `AF003`'s general requirement that any write/destructive tool declare explicit permissions.

### Unrestricted shell execution

A tool that can run arbitrary shell commands with no declared constraints. **Mitigated** structurally: [`AF002`](../../packages/policy/src/policies/af002-no-unrestricted-shell-tools.ts) flags a `function`/`customPlugin`/`mcp` tool whose handler/plugin/operation name indicates shell execution and has no declared `permissions`. As with arbitrary code execution above, this is a pre-execution structural check — there is no tool-execution runtime yet for it to gate directly.

### Poisoned imported projects

A project composed via `$ref`/auto-discovery from an untrusted source smuggling in a malicious resource. **Partially mitigated**: an imported file is bound by exactly the same project-root sandbox and the same policy pack (`AF001`–`AF015`) as hand-authored content — it can't escape the sandbox, and if it inlines a secret, an unrestricted shell tool, or an unpermissioned write tool, the same policies catch it at validation time. What's _not_ yet supported is a separate trust tier for imported vs. authored content (e.g. "apply stricter policy severity to anything pulled in from outside this repository") — everything is validated equally today.

### Unsafe YAML behavior

YAML's historical footguns: custom tag execution, unbounded anchor/alias expansion, implicit type coercion surprises. **Mitigated**: parsing goes through the `yaml` npm package's `parseDocument`, which does not execute custom tags or construct arbitrary objects the way some other ecosystems' "unsafe load" functions do; duplicate keys are rejected (`uniqueKeys: true`) rather than silently overwriting.

### Denial-of-service through recursive references

A `$ref` chain designed to recurse until the process exhausts memory or its call stack. **Mitigated**: reference cycle detection (`AGF1003`) and a maximum reference depth (`AGF1004`, default 32) both stop resolution rather than recursing indefinitely. Proven against a real on-disk fixture in [`specifications/v1alpha1/security/recursive-reference-exhaustion`](../../specifications/v1alpha1/security/recursive-reference-exhaustion).

### Workflow infinite loops

A workflow graph cycle with no bound on iteration count. **Mitigated**: `@agentform/ir`'s graph cycle detection rejects any cycle not passing through a `loop` node (`AGF3007`), and a `loop` node's `maxIterations` is schema-required to be a positive integer — [`AF005`](../../packages/policy/src/policies/af005-workflow-loops-require-limits.ts) re-verifies the same invariant at the policy layer as defense in depth against a `PolicyContext` built from data that bypassed schema validation.

### Cost exhaustion

A run that spends far more (in token or dollar cost) than intended. **Partially mitigated**: the schema supports per-agent `limits.maxCostUsd`/`limits.maxSteps` and per-model `rateLimits`/`costMetadata`, and the structural size caps (`AGF3016`–`AGF3018`) bound the worst-case shape of a single workflow. Phase 10 adds `maximumCost`/`maximumAverageCostUsd` assertions and threshold gates (`docs/evaluation-reference.md`) that catch a cost regression in `agentform test` against the deterministic mock engine — genuinely useful for "did this change to the spec make a mocked run cost more," but that's a test-time check against author-supplied mock cost figures, not run-time enforcement against a real bill. Phase 11's `agentform apply` now runs these same assertions as a genuine blocking gate (a failing smoke test exits 9 and persists nothing), so a cost regression caught by a dataset now actually prevents that specification from being applied, not just reported by `agentform test` in isolation — still against mocked figures, since there is still no run-time cost _enforcement_ against a real provider or bill. Nothing executes an agent against a real provider yet, so nothing can overspend yet — that's the deploy/runtime work still ahead (no adapter implements `deploy` yet).

### Tool-call amplification

A structure that causes a small input to trigger an unbounded or exponential number of tool calls (unbounded retries, unbounded fan-out). **Partially mitigated**: `retry.maxAttempts` and `limits.maxSteps` are schema-supported per agent; workflow node/edge caps bound structural fan-out; loop nodes require `maxIterations`; Phase 10's `maximumToolCalls`/`maximumRetries` assertions (`docs/evaluation-reference.md`) let a test dataset catch a scenario where a mocked run calls a tool, or retries, more than expected — and, as of Phase 11, `agentform apply` runs these same assertions as a blocking smoke-test gate before persisting any state, the same upgrade cost exhaustion got above. As with cost exhaustion, _counting and stopping_ excessive calls against a real provider at run time needs a runtime that doesn't exist yet.

## Verifying these protections

- `packages/parser/src/refs.test.ts` and `packages/parser/src/discover.test.ts` — reference cycle, max depth, max file size, unsafe path, in-memory filesystem.
- `packages/parser/src/security-fixtures.test.ts` — the same protections proven against real on-disk fixture projects rather than the in-memory test harness.
- `packages/ir/src/semantic/limits.test.ts` — workflow node/edge/expression-length caps.
- `packages/policy/src/policies/*.test.ts` — every built-in policy, including the redaction and mandatory-override tests in `packages/policy/src/evaluate.test.ts` and `packages/policy/src/redact.test.ts`.
- `packages/state-local/src/lock.test.ts` and `sqlite-state-backend.test.ts` — lock contention/staleness, atomic transactions, crash recovery, all against real temp-directory SQLite files.
- `packages/planner/src/plan-file.test.ts` — plan file tamper detection, including an attempt to also edit the recorded hash.
- `packages/compiler/src/secret-scan.test.ts` and `packages/compiler/src/compile.test.ts` — the generated-code secret scan and blocking-incompatibility gate, including a fixture that would otherwise leak a secret-shaped value.
- Every `packages/adapter-*/src/adapter.test.ts` (openai, langgraph, microsoft, google-adk, autogen, crewai) — deterministic generation (byte-identical repeat `generate()` calls) and isolation (a prompt-only change alters only that agent's file), the two properties `AF015`/manifest hashing depend on.
- `apps/cli/src/cross-target.test.ts` — the same portable spec compiled against all six real adapters, proving the blocking-incompatibility gate and secret scan aren't openai/langgraph-specific behavior.
- `packages/evaluator/src/results-record.test.ts` and `gate-status.test.ts` — evaluation-results tamper detection (mirroring the plan-file tests exactly) and the never-run/stale/failed/passed classification.
- `apps/cli/src/lib/report-redaction.test.ts` and `apps/cli/src/commands/test.test.ts`'s "secret redaction" describe block — a secret-shaped value in a mocked tool call is proven absent from console, `--json`, and `--junit` output, against a real `agentform test` run.
- `apps/cli/src/commands/apply.test.ts` — atomic persistence on failure (a failing smoke test persists nothing and releases the lock), tampered/stale plan-file rejection, policy failure blocking even with `--auto-approve`, critical-change confirmation.
- `apps/cli/src/commands/rollback.test.ts` — audit-history preservation (three operations, three preserved rows), a genuine execution-phase failure (an unreadable snapshot) leaving history/resources untouched and releasing the lock, declined/non-interactive confirmation changing nothing.
- `apps/cli/src/commands/destroy.test.ts` — unconditional confirmation, a declined/non-interactive destroy destroying nothing and releasing the lock, the "cannot be recovered" warning naming real on-disk paths, audit-history preservation.
- `packages/state-local/src/sqlite-state-backend.test.ts` — `withTransaction` commit/rollback, `readBackupSnapshot`'s isolation from the live database, `recordDriftStatus`'s narrow mutation.
- `packages/planner/src/destroy-plan.test.ts` — destroy-plan risk classification and reverse-dependency ordering, independent of `comparePlan`.
- `apps/cli/src/commands/import.test.ts`, `packages/adapter-openai/src/inspect-existing.test.ts`, `packages/adapter-langgraph/src/inspect-existing.test.ts`, `apps/cli/src/lib/import-generated.test.ts` — every recognizer's honesty properties: an unrecognized project reports `recognized: false` rather than guessing, every recognized result carries a non-1.0 confidence unless it's genuinely exact (the generated-project recognizer's identity recovery), and no recognizer ever executes anything from the project it's inspecting.

## Updating this document

Add a section (or update a status label) whenever a phase changes what's actually enforced — a status of "Planned" that quietly becomes true without this document being updated is itself a documentation-drift risk. Phase 8 moved "Generated-code tampering" from Planned to Partially mitigated and added the blocking-incompatibility and generated-code-secret-scan rows above; it deliberately left "Generated-code tampering" at Partially (not Mitigated) since there's still no automated verify command, and left "Prompt injection" at Planned since Phase 8's guardrail stubs are structural scaffolding, not real runtime filtering. Phase 9 extended the same "Generated-code tampering"/"Blocking incompatibility"/"Generated-code secret scan" rows to cover Microsoft/Google ADK/AutoGen/CrewAI rather than opening new ones — every mitigation described above (deterministic generation, manifest hashing, `hasBlockingIncompatibility`, `scanForSecretLeaks`) is implemented once in `@agentform/compiler`'s adapter-agnostic `compile()` pipeline, not per-adapter, so extending coverage to four more targets confirmed the design rather than requiring new controls. Phase 10 added the new "Evaluation gate tampering" threat and extended "Secret leakage"/"Cost exhaustion"/"Tool-call amplification" for the evaluation engine — `.agentform/test-results.json` reuses the exact same tamper-evidence pattern as a `.afplan` file, deliberately kept at "Mitigated (detection only)" rather than "Mitigated" outright since, like plan-file verification, nothing yet reads it at blocking (`error`) severity; that noted this would move once `apply` existed to actually refuse to proceed.

**Phase 11 is that milestone.** "Plan tampering" moved from "detection only" to fully Mitigated — `agentform apply` now calls `verifyPlanFile` for real, and separately rejects a stale (untampered but outdated) plan via fingerprint comparison. "Evaluation gate tampering" stayed at "Mitigated (detection only)" for `plan`'s advisory reporting (that part is unchanged, by design — ADR-0011), but gained a new "Evaluation gate enforcement (blocking)" row: `agentform apply` doesn't read the tamper-evident record at all, it re-runs smoke tests fresh and genuinely blocks on failure, which sidesteps record-tampering as a concern for `apply` specifically rather than "fixing" the advisory path. "State tampering" was extended for `withTransaction` (atomic multi-write persistence, not just atomic per-migration) and the fact that the state lock is now actually held across real mutations, not only modeled. Two new threats were added: "Rollback/backup integrity" (why `agentform rollback` can restore state without ever being able to erase audit history — a structural guarantee from never calling `restoreBackup`) and "Unintended or silent destruction" (why `agentform destroy`'s confirmation is unconditional and distinguishes recoverable state from unrecoverable artifacts/deployments). "Cost exhaustion" and "Tool-call amplification" were both extended to note that their evaluation-dataset checks are no longer purely advisory — `agentform apply` runs them as a real blocking gate. See ADR-0012 (apply, drift) and ADR-0013 (rollback, destroy) for the full design reasoning behind each of these.
