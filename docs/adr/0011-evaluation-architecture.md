# ADR-0011: Evaluation architecture — deterministic mock execution, structural assertions, and advisory (not yet blocking) gates

## Status

Accepted

## Context

Phase 10 (`temp/instruction.md`'s Phase 10 objectives, and the spec's own required-ADR list item "Evaluation architecture") requires `@agentform/evaluator` and `@agentform/runtime`, wired into a new `agentform test` command, plus "integrate evaluations into plan/apply gates." Three real constraints shaped the design: (1) no Agentform runtime executes a real model or tool anywhere in the codebase yet — Phase 11 (`apply`) is the first phase that will — so "run a test" cannot mean "call a real provider" without inventing that infrastructure a phase early; (2) the build spec's own evaluation vocabulary is described at three different altitudes across §6.8, §17, and Phase 10's own checklist that don't perfectly agree with each other; (3) "apply" doesn't exist yet, so "failed evaluation gates block production apply" (Phase 10's own acceptance criterion) cannot be literally true today — something has to give without silently dropping the requirement.

## Decision

- **A deterministic, fully offline mock execution engine (`@agentform/runtime`) walks the real IR workflow graph, using real IR data for every structural decision, but never calls a real model, tool, or API.** `runWorkflow(ir, scenario)` uses an agent's actual `retry.maxAttempts` for retry bounds and a loop node's actual `maxIterations` for the loop safety valve — genuine IR data, not fabricated defaults — while every effect (a tool's return value, whether a human approval is granted, which edge a branch takes) comes entirely from the test author's own scenario declaration. This mirrors every framework adapter's own `when`-condition stub precedent (ADR-0009/0010): no real expression evaluator exists anywhere in Agentform, so a workflow's `when:` strings are never evaluated for real here either — a branching or looping scenario must explicitly say which way it goes via `nodes.<id>.next`.
- **The assertion vocabulary is 16 types: the 14 named in Phase 10's own "minimum evaluators" checklist, plus `nodeNotVisited` and `fieldRange` from §17/§6.8's fuller vocabulary that checklist happens to drop.** The three altitudes (§6.8's domain-model list, §17's one concrete YAML example, Phase 10's checklist) disagree on naming and completeness; Phase 10's checklist was treated as the primary implementation contract since it's the most specific and literal, but two cheap, natural complements from the other two lists were included anyway (see `packages/evaluator/src/assertion.ts`'s own doc comment) rather than mechanically implementing exactly 14 and ignoring real vocabulary the spec clearly intends.
- **Datasets support `.jsonl` (one JSON test case per line — §17's own canonical example format) plus `.json`/`.yaml` (a bare array, or an object with a `tests:` array — §17's inline YAML block shape).** Both are real formats the spec itself demonstrates, not an invented third option.
- **Threshold gates (`spec.evaluations.thresholds`) actively gate exactly three keys — `taskSuccess`, `policyViolations`, `maximumAverageCostUsd` — the ones the build spec's own canonical example uses.** The schema stays open to any string key (forward compatible), but an unrecognized key is reported (`recognized: false`) rather than silently ignored (§12) or silently accepted as if it did something. Inventing gating behavior for keys the spec never actually defines a formula for would be guessing, not implementation.
- **`.agentform/test-results.json` is a tamper-evident record deliberately built to mirror `@agentform/planner`'s `.afplan` `PlanFile` design exactly** — a content hash over the rest of the record, computed with the same canonicalization `computeContentHash` already uses, verified by a parse function that returns `{ valid: false, error }` rather than throwing. Reusing an already-battle-tested pattern (with its own test suite already proving round-trip, tampered-field, and tampered-hash cases) is lower-risk than inventing a new integrity scheme for what is structurally the same problem: "did a past decision's recorded output actually happen, unmodified, for the state that exists right now."
- **`checkEvaluationGateStatus` (never-run/stale/failed/passed) is a pure function in `@agentform/evaluator` with no knowledge of "production."** `@agentform/evaluator` does not depend on `@agentform/policy` anywhere (the existing `EvaluationContext.policyPassed` pattern already established this boundary for `policyResult` assertions) — deciding _whether_ gate status matters for a given environment is left entirely to the CLI layer, which already depends on both packages and is where `isProductionEnvironment` already lives (used identically by `AF008`/`AF013`).
- **`agentform plan` surfaces gate status as `AGF6001`/`AGF6002`/`AGF6003` — always `warning` severity, never `error`, and only for a production `runtime.environment` that also declares at least one dataset or threshold.** This is the honest resolution of "integrate evaluations into plan/apply gates" and "failed evaluation gates block production apply" a phase early: `apply` doesn't exist yet (Phase 11), so nothing in this phase can actually _block_ a deployment — claiming otherwise would be fabricating enforcement that isn't real. `agentform status` reports the same status unconditionally (any environment) as part of its existing read-only summary, replacing a previously-hardcoded "evaluation engine is not implemented" string that would otherwise have gone stale the moment this phase shipped. Production-but-nothing-declared is deliberately silent here (`AF008` already reports that case as a policy error) rather than doubling up on a less specific "never run" warning for a spec with nothing to run in the first place.
- **`agentform test --live` is a recognized flag that's rejected immediately (exit 2, a clear message) rather than partially implemented or silently ignored.** There is no credential-loading or real-provider-calling infrastructure anywhere in Agentform yet (`@agentform/secrets-env` remains an unimplemented skeleton) — building that is a substantially larger, separate undertaking than the deterministic engine, and a flag that silently behaved like `--live` was absent would be a worse outcome than one that says so.
- **Test reports (console, `--json`, `--junit`) are passed through the same `SECRET_PATTERNS`/`redactSecretValue` `@agentform/policy`'s `AF001` uses, applied to the fully rendered report text.** This was not in the original Phase 10 task breakdown but follows directly from the phase's own acceptance criterion ("test reports contain no secrets"): a dataset's mocked tool call content is author-controlled test-fixture data — commonly captured from a real API response while writing a mock — that `AF001` never scans, since dataset files are loaded separately from the specification document policy evaluation walks. Left unaddressed, a real credential pasted into a fixture would flow verbatim into `--junit` output specifically designed to feed CI dashboards that archive what they're given.
- **"Adapter-generated smoke tests" (one of Phase 10's stated objectives) is satisfied by Phase 8/9's existing per-adapter toolchain verification and `apps/cli/src/cross-target.test.ts`, not by new code-generation work in this phase.** Every adapter's generated project is already verified against its real toolchain (`pip`, `dotnet build`, real script execution), and one portable specification already compiles against all six adapters. Translating evaluation datasets into framework-native test files embedded in a generated project would be a new code-generation capability neither this phase's required packages (`@agentform/evaluator`, `@agentform/runtime`) nor any CLI flag/acceptance criterion in the Phase 10 section actually calls for.

## Alternatives considered

- **A real (even minimal) expression evaluator for `when:` conditions, so branching/looping could be decided by the scenario's `input` rather than requiring an explicit `nodes.<id>.next` override**: rejected — this would be the first real expression evaluator anywhere in Agentform, a scope increase far beyond "evaluation engine," and would only benefit test authoring ergonomics, not correctness; every adapter already generates a `when`-stub for a human to fill in, so a scenario explicitly stating its own branch choice is consistent with that same "structural only" discipline, not a workaround for a missing feature.
- **Gating `agentform plan`'s evaluation diagnostics at `error` severity for production, blocking the exit code today**: rejected — there is no `apply` yet for a blocked `plan` to actually protect; an `error`-severity diagnostic with nothing real behind it to prevent would be a false sense of enforcement, worse than an honest warning.
- **Implementing threshold gating for every key a user might plausibly write (`maximumLatency`, `maximumRetries`, etc.) by inventing a formula for each**: rejected — the build spec defines a formula (via its own example) for exactly three keys; inventing gating behavior for the rest would be guessing at intent the spec never states, when the existing per-test-case assertion vocabulary (`maximumLatency`, `maximumRetries`, `maximumCost`) already covers the same concerns at the test-case level.
- **A separate secret-redaction mechanism specific to `@agentform/evaluator`**: rejected in favor of reusing `@agentform/policy`'s existing `SECRET_PATTERNS`/`redactSecretValue` from the CLI layer — `@agentform/evaluator` deliberately doesn't depend on `@agentform/policy` (see above), and the CLI already depends on both, making it the natural integration point without adding a new cross-package dependency for a handful of shared constants.
- **Building `agentform test --live` against at least one real provider (e.g. OpenAI) now, rather than rejecting the flag outright**: rejected as disproportionate scope for this phase — it would require credential loading, network-call error handling, and a first real runtime execution path, none of which exist yet and all of which are legitimately larger undertakings than the rest of this phase combined.

## Consequences

- `agentform test` is fully deterministic and requires no network access or credentials, satisfying "offline tests require no external API" without qualification — but it also cannot yet catch anything a real provider's actual behavior would surface (a real model refusing a request, a real API's actual latency/cost, a real tool's real failure mode). That gap is inherent to "no live execution yet," not a bug.
- `agentform plan`/`agentform status` both now depend on `@agentform/evaluator` in addition to `@agentform/policy`/`@agentform/planner` — a small, one-directional addition consistent with the CLI's existing role as the integration layer across every domain package.
- A future `agentform apply` (Phase 11) has a ready-made, already-tested integration point: `checkEvaluationGateStatus` plus the existing `AGF6xxx` codes need only a severity change (and a caller that actually enforces it) to become a real blocking gate, rather than needing new design work at that point.
- Threshold gates and assertions overlap in what they can express for cost/latency/retries (a per-test-case assertion vs. an aggregate run-level gate) — this is intentional layering (a single test's cost spike vs. the whole dataset's average), not accidental duplication, but it does mean an author has two different places to express a similar-sounding constraint.

## Security impact

The deterministic mock execution engine never calls a real model, tool, HTTP endpoint, or subprocess — every "effect" in a run is data the test author already declared, so a hostile dataset cannot use `agentform test` to reach a real system. Dataset files are loaded through the same `resolvePathWithinRoot` sandbox as every other file reference. `.agentform/test-results.json` is tamper-evident the same way `.afplan` is. New in this phase: `redactSecretsFromReport` closes a real gap where a dataset's author-supplied mock content could otherwise leak a captured credential into `agentform test`'s console/`--json`/`--junit` output. `docs/security/threat-model.md` is updated alongside this ADR with a new "Evaluation gate tampering" entry and extensions to "Secret leakage"/"Cost exhaustion"/"Tool-call amplification".

## Migration impact

`@agentform/evaluator` and `@agentform/runtime` are new packages; `agentform test` is a new command. `agentform plan`'s diagnostics array gains up to one new warning-severity entry (`AGF6001`/`AGF6002`/`AGF6003`) under the stated conditions — additive, never changes `plan`'s exit code. `agentform status`'s `evaluationStatus` field changes value (from a hardcoded "not implemented" string to a real computed status) but not shape — any consumer of `--json status` parsing that field as a string is unaffected. No existing package's public API changes.
