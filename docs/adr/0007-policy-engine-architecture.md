# ADR-0007: Policy engine architecture — single effective severity, structural-only mandatory protection, one config file

## Status

Accepted

## Context

Phase 6 (`temp/instruction.md` §6.7, §16, and Phase 6's own objectives) requires a policy engine with a built-in pack of at least 15 policies, configurable severity "only within allowed organizational bounds," and an explicit acceptance criterion that "mandatory policies cannot be bypassed with CLI flags." The spec gives the `PolicyResult` interface shape and the list of 15 policy IDs/names, but not: how a `PolicyDefinition`'s own severity interacts with what an individual check finds, how "cannot be bypassed" is actually enforced in code rather than just documented as a rule, what a policy configuration file looks like or where it lives, or how three of the fifteen policies (`AF005`, `AF007`, `AF014`, `AF015`) should behave when the thing they check either can't structurally fail (schema already guarantees it) or doesn't exist yet (no state engine, no compiler). Those are the decisions this ADR records.

## Decision

- **A policy has exactly one effective severity; a finding never carries its own.** `PolicyFinding` (what a `check` returns) has no `status` field — only `message`/`resourceAddress`/`remediation`. `evaluatePolicies` derives every finding's reported status (`fail` or `warn`) from the *policy's* effective severity (its `defaultSeverity`, or a validly-applied override). This was a deliberate simplification from an earlier draft where findings carried their own `'fail' | 'warn'` status: that would let a single policy's findings disagree with its own configured severity, which is confusing and serves no real use case here (§16's `PolicyResult` doesn't ask for per-finding severity independent of the policy).
- **Mandatory-policy protection is structural, not conventional.** `evaluatePolicies` rejects *any* severity change to a mandatory policy's override, full stop — not "reject weakening but allow tightening," which would be a more permissive rule nobody asked for and that adds a branch with no test-observable benefit. A same-value override (a no-op) is silently accepted even for a mandatory policy, since nothing is actually changing. This is what makes "cannot be bypassed with CLI flags" true in code: there is no code path, CLI flag or otherwise, that can change a mandatory policy's effective severity — the enforcement lives at the one place all overrides funnel through, not at the CLI layer where it would be one omitted check away from a bypass.
- **Weakening a non-mandatory policy requires a non-empty, non-whitespace `justification`; tightening or no-op never does.** Read literally from §16's minimum test list ("Policy exception with justification"). Justification is stored nowhere beyond the rejection/acceptance decision itself today — there is no audit log yet (that's observability-phase scope, §18) — but requiring it now means the *shape* (a `PolicyOverride.justification` field) already exists for a future phase to persist without a breaking change.
- **Exactly one policy configuration file (`agentform.policy.yaml`), found by fixed name at the project root, no CLI flag to relocate it.** Mirrors the already-established `environments/<name>.yaml` convention (absent is fine, present-but-malformed is a blocking diagnostic). The "no flag" part is deliberate: a `--policy-config <path>` flag would still respect every override rule above, but it's one more surface to reason about for zero requested capability, and every additional flag around policy configuration is one more thing a future contributor could add a "just this once" bypass to. Keeping exactly one, fixed, versioned-alongside-the-project source of overrides is what makes "mandatory policies cannot be bypassed with CLI flags" trivially true rather than true-until-someone-adds-a-flag.
- **`AF005` and `AF007` are real checks even though the schema already makes their target condition hard or impossible to violate.** `AF005` (loop `maxIterations` must be positive) re-verifies an invariant `loopNodeSchema` already enforces via `z.number().int().positive()` — proven via a test that deliberately constructs a `PolicyContext` bypassing schema validation, showing the check has real teeth as defense in depth, not dead code. `AF007` (model `provider` must be non-blank) is not actually redundant: `z.string().min(1)` accepts a single-space string, so `AF007` closes a real gap the schema leaves open.
- **`AF014`/`AF015` are honest, mandatory, always-passing placeholders**, not implemented as no-ops that skip registration. There is no state engine (Phase 7) or compiler (Phase 8) yet for them to inspect, so their `check` returns `[]` unconditionally — but they're registered in `BUILTIN_POLICIES` as `mandatory: true` *now*, specifically so a project's `agentform.policy.yaml` committed today cannot contain an override that silently disables the real check the moment Phase 7/8 gives them something to enforce. The alternative — omitting them until they have real logic — would let exactly that override slip in unnoticed.
- **`AF012` checks declaration, not enforcement, and says so.** There is no organization-level network-destination allowlist registry in the schema, and no HTTP-tool execution runtime to enforce a destination against one at request time. Rather than either skip the policy entirely or overreach into designing a new config surface not asked for, `AF012` enforces the one thing that's both checkable today and a real prerequisite for allowlisting later: an `http`/`openapi` tool must explicitly declare `networkDestination` rather than leaving it unconstrained. Documented explicitly as a scoping decision (in `docs/policy-reference.md`'s Scope section) rather than left to look like more coverage than it provides.

## Alternatives considered

- **Findings carry their own severity, independent of the policy's**: rejected — see Decision above; adds a confusing dual-source-of-truth for severity with no requested use case.
- **Allow overriding a mandatory policy's severity upward (tightening) but not downward**: rejected — nothing in the spec asks for this, all fifteen built-in policies default to `error` except `AF006`, so the only realistic "tightening" case is `AF006` warning→error, which is a one-policy edge case not worth a special-cased branch; "mandatory means fixed, period" is simpler and easier to audit.
- **`--policy-config <path>` CLI flag**: considered and rejected — see Decision above.
- **Skip `AF014`/`AF015` entirely until Phase 7/8**: rejected — registering them now as mandatory placeholders is strictly safer than adding them later, for the reason given above.
- **A richer network-allowlist config surface for `AF012` (e.g. a new schema field or a separate allowlist file) now**: rejected as scope creep for Phase 6 — no phase has asked for this shape yet, and inventing one risks conflicting with how a later phase (deployment/state) actually wants to model it.

## Consequences

- Any future built-in policy that needs to report *varying* severity for different findings within the same evaluation would require revisiting the single-effective-severity design — no such policy exists among the 15 required ones, so this is a real but currently-hypothetical constraint, not a known gap.
- `PolicyOverride.justification` is validated and stored transiently (used once, at evaluation time, to decide whether to accept a downgrade) but not persisted or surfaced anywhere beyond that decision — a future observability/audit phase that wants "who approved this exception and when" will need to add that persistence; the shape is ready, the storage isn't.
- `agentform.policy.yaml`'s fixed location means a monorepo with multiple Agentform projects needs one such file per project root, exactly like `environments/` — consistent, but worth knowing if a future phase wants shared organization-wide policy config (out of scope today; §6.7's "organization-level policy packs" is explicitly deferred).

## Security impact

This ADR is largely about security enforcement: the structural (not conventional) mandatory-policy protection and the single-config-file design are both direct responses to §16's "mandatory policies cannot be bypassed" acceptance criterion. `AF001`'s findings are redacted via `redactSecretValue` before ever reaching a diagnostic, log, or test snapshot. See `docs/security/threat-model.md` for the full cross-package picture.

## Migration impact

None — this is the initial policy engine implementation. `BuildIRResult` and `apps/cli`'s `LoadAndBuildResult` both gained an additive `application`/`ir.application`-adjacent field (non-breaking) so `validate` can hand the schema-validated document to the policy engine without re-validating it.
