# ADR-0014: Module registry architecture — a local, signable registry resolved before schema validation

## Status

Accepted

## Context

Phase 12 (§Phase 12 "module concept") asks for a way to package and reuse agentic-system resources across projects — a module analogous to a Terraform module: versioned, inputs/outputs, resolvable from a registry, safely mergeable into a consuming project. Nothing in Phases 1-11 needed this; every resource so far was either authored inline or pulled in via `$ref`/auto-discovery from files the project itself owns. A module is different in kind: it's authored and versioned independently of any one project, published once, and consumed by many. That requires three things the existing parser/schema/IR pipeline has no reason to have built: a place modules live (a registry), a way to tell a genuine module from a tampered or malicious one, and a merge step that turns a project's `modules:` declaration into real resources without becoming a second, weaker validation path alongside the one every hand-authored resource already goes through.

## Decision

- **Modules are their own top-level document kind, validated by the same resource schemas everything else uses.** `moduleDefinitionSchema` (`packages/schema/src/module.ts`, `apiVersion: agentform.dev/v1alpha1`, `kind: AgentformModule`) has a `spec` whose `models`/`tools`/`agents`/`workflows`/`memory` fields are the literal same `modelSchema`/`toolSchema`/`agentSchema`/`workflowSchema`/`memorySchema` the top-level application document uses. A module's `agents.triage` and a project's inline `spec.agents.triage` are validated by identical code — there is no parallel, looser module-specific resource schema to keep in sync or accidentally under-validate.
- **The registry is a local filesystem store, not a network service.** `publishModule`/`resolveModule`/`listModules` (`packages/registry/src/local-registry.ts`) read and write `<registryRoot>/<source>/<version>/{module.yaml,manifest.json}` directly; `registryRootFor()` (`apps/cli/src/lib/registry.ts`) defaults to `~/.agentform/registry`, the same per-machine shared-cache convention npm/pnpm already use for their own package stores. A team shares modules today by sharing a filesystem location (network drive, synced directory, CI artifact) that every consuming machine points `AGENTFORM_REGISTRY_ROOT` at — building and operating an actual registry server is out of scope for this phase and not something the build spec asks for.
- **Tamper detection is structural, not optional.** `publishModule` records a `contentHash` (`@agentform/ir`'s `computeContentHash`, the same canonicalization `.afplan` files and generated-code manifests already use) in `manifest.json` at publish time; `resolveModule` recomputes that hash from the on-disk `module.yaml` on every resolve and throws if it no longer matches. There is no "trust the manifest" path — a module's integrity is checked every time it's used, not just at publish time.
- **Signing is opt-in per module and per consuming machine, layered on top of the content hash rather than replacing it.** `signContentHash`/`verifyContentHashSignature` (`packages/registry/src/signing.ts`) use Node's built-in `node:crypto` Ed25519 support — no new dependency for something the runtime already provides. A publisher signs (or doesn't) by passing a private key to `publishModule`; a consumer trusts (or doesn't) by setting `AGENTFORM_REGISTRY_TRUSTED_KEY`. Neither side is required, and the two are independent: an unsigned module resolved with no trusted key configured behaves exactly as it did before signing existed.
- **An unsigned module is a warning, not an error, even when a trusted key is configured — but a signed-and-invalid module is always an error.** `resolveProjectModules`'s `verifySignature` treats "no signature, but I have a trusted key" as `AGF7006` at `warning` severity (provenance can't be confirmed, but nothing indicates tampering either) and "has a signature, but it doesn't verify" as the same code at `error` severity (positive evidence of tampering or a wrong key). Requiring every module to be signed once any trusted key exists would make signing all-or-nothing per machine, which isn't how this ecosystem's few-if-any modules will actually be authored at this phase; treating a bad signature as fatal is not a similar judgment call — it's the one case that specifically indicates something is wrong.
- **Module resolution runs in the pipeline before schema validation, not after.** `apps/cli/src/lib/pipeline.ts`'s `loadAndBuildIR` calls `resolveProjectModules` between `loadProject` (parsing, `$ref`/overlay/variable resolution) and `buildIR`. This ordering is the load-bearing decision in this ADR: it means a module's contributed resources are validated by `agentSchema`/`toolSchema`/etc., checked by every `AF0xx` policy, and checked by IR semantic validation exactly as any inline resource would be. A module cannot smuggle in something that bypasses validation by virtue of arriving through a different code path.
- **`spec.modules` itself survives resolution in the returned document, unlike `variables`/`locals`.** `@agentform/parser`'s `interpolateDocument` strips `variables`/`locals` once they've been substituted, since nothing downstream needs to inspect them again. `resolveProjectModules` deliberately leaves `spec.modules` in place — it's real, inspectable project metadata (which modules this project depends on, at which versions) that `agentform inspect`/`agentform lockfile` and a human reader legitimately want to see, not a resolution-only scratch section.
- **A `${input.NAME}` interpolator separate from `@agentform/parser`'s `${env.*}`/`${var.*}`/`${local.*}` interpolation.** `substituteInputs` (`packages/registry/src/input-substitution.ts`) is small and self-contained rather than a new case added to the parser's existing interpolator. The parser's interpolation is exercised by every project that has ever used this codebase and is deliberately not touched lightly; adding a module-specific namespace to it would mean either extending a fixed, well-tested set of namespaces for a feature most projects don't use, or risking regressions in interpolation behavior every project depends on. A duplicate ~60-line interpolator is cheaper than that risk.
- **Resource collisions resolve by "whichever was declared first wins," matching the parser's existing auto-discovery precedent.** When a module's resource identifier collides with one already in the accumulator — inline, or from an earlier-declared module — `mergeModuleResources` keeps the existing one and reports `AGF7005`. This is the same rule `@agentform/parser`'s file-auto-discovery already uses for colliding file-discovered resources; introducing a different collision rule (e.g. "last wins," or "always an error") for modules specifically would be an inconsistency with no motivating requirement.
- **`agentform.lock` pins resolved modules by content hash, mirroring `.afplan`'s own tamper/drift model.** `buildLockfile` (`packages/registry/src/lockfile.ts`) records each resolved module's `source`/`version`/`contentHash`/`signatureVerified`; `agentform lockfile --check` compares a fresh resolution against what's on disk without writing, the same "detect drift, don't auto-correct it" posture `agentform drift` already uses for deployed state. `agentform validate`/`apply` do not yet cross-check the lockfile automatically — that would need a decision about whether lockfile drift should block those commands outright, which the build spec doesn't specify and is deferred rather than guessed at.
- **`AGF7xxx` is a new, dedicated diagnostic range**, following the existing per-package convention (`AGF1xxx` parser, `2xxx` schema, `3xxx` ir, `4xxx` policy, `5xxx` compiler, `6xxx` evaluator) rather than reusing another range or inventing an unstructured one.

## Alternatives considered

- **A remote/HTTP registry service from the start**: rejected for this phase — nothing in the build spec's Phase 12 scope asks for a hosted service, and building one would introduce a real network dependency, authentication, and availability concerns for a feature whose core value (packaging and versioning reusable resources) doesn't require one. A local, shareable-by-filesystem registry is a strict subset of what an HTTP registry would offer and can be layered under one later without changing `resolveModule`'s contract (source/version in, definition out).
- **Making module signing mandatory**: rejected — it would block every current use of modules (nothing in this ecosystem has published a signed module yet) for a guarantee only meaningful once there's an actual distribution channel where signature verification matters. Optional signing, checked strictly when configured, gets the safety property without the adoption cliff.
- **Extending `@agentform/parser`'s interpolator with an `${input.*}` case**: rejected, per the Decision section above — too much risk to a fixed, heavily-relied-on code path for too little shared logic (the two interpolators don't actually share much beyond "find `${...}` and replace it").
- **Running module resolution after schema validation, validating a module's raw resources separately inside `@agentform/registry`**: rejected — this would create exactly the second, potentially-drifting validation path the Decision section rules out. Running it before validation means `@agentform/registry` never needs its own copy of resource-shape rules; it only needs to know how to merge.
- **"Last module wins" or "error on any collision" instead of "first wins"**: rejected in favor of consistency with the parser's existing file-discovery collision rule — introducing a second, different collision policy for a conceptually similar situation (two sources contributing a same-named resource) would be a needless inconsistency for a reader trying to build one mental model of "what happens when two things declare the same id."
- **Blocking `agentform validate`/`apply` on lockfile drift automatically**: deferred, not rejected outright — doing this without the build spec specifying the intended behavior (warn? block? only in CI?) risks guessing wrong about a workflow decision that affects every project using modules. `agentform lockfile --check` exists today as an explicit, opt-in step (e.g. a CI job) until that's specified.

## Consequences

- `apps/cli/src/lib/pipeline.ts`'s `LoadAndBuildResult` gained a `resolvedModules` field every caller of `loadAndBuildIR` now receives, whether or not the project being loaded declares any modules (an empty array when it doesn't) — a small but permanent shape change to the pipeline's return type.
- Every future resource collection added to the schema (a hypothetical new top-level kind) that should be modularizable needs a corresponding entry added to `resolve-project-modules.ts`'s `MERGED_COLLECTIONS` and to `moduleSpecSchema` — module support is not automatic for new resource kinds.
- `@agentform/registry` depends on `@agentform/schema`, `@agentform/ir`, and `@agentform/plugin-sdk` (for the parallel plugin-registry metadata shape) — a new package dependency edge into the pipeline that didn't exist before Phase 12.
- A project that never declares `spec.modules` pays no cost: `resolveProjectModules` returns the document unchanged with empty diagnostics/`resolvedModules` the moment `spec.modules` is absent, before touching the filesystem at all.
- The local-registry-only design means multi-machine/multi-developer module sharing is a filesystem-sharing problem for the team to solve themselves (shared drive, synced folder, CI cache) until a hosted registry is built — this is a known, deliberate scope limit of this phase, not an oversight.

## Security impact

Module resolution introduces a new trust boundary: a project can now pull in resources it didn't author inline. Three structural properties bound that risk. First, a resolved module's resources are subject to the exact same policy pack (`AF001`-`AF015`) and IR semantic validation as any inline resource — a module cannot declare an unrestricted shell tool, an inline secret, or an ungated destructive action any more successfully than hand-written YAML could, because both paths converge before those checks run. Second, integrity is checked on every resolve via content-hash recomputation, not trusted from a cached manifest — a `module.yaml` edited on disk after publishing (accidentally or maliciously) is caught as `AGF7002` rather than silently used. Third, signature verification, when a trusted key is configured, turns a positive-but-invalid signature into a hard error rather than a soft warning, so a wrong-key or tampered-content scenario a publisher intended to be caught actually is. The residual risk this phase leaves open and unmitigated is the same one file references have always carried: a _registry itself_ being on compromised or malicious shared storage is equivalent to any other untrusted file source reachable from the project root, with no additional registry-specific protection beyond what's described here. `docs/security/threat-model.md` gains a new "Poisoned or malicious modules" entry alongside this ADR.

## Migration impact

New packages: `@agentform/registry` (new). New schema surface: `spec.modules` on the application document (optional, backward compatible — an existing project with no `modules:` field validates and resolves exactly as before), plus the new `AgentformModule` document kind. New CLI command: `agentform lockfile [--environment] [--check]`. `apps/cli/src/lib/pipeline.ts`'s `loadAndBuildIR` signature is unchanged, but its return type gains `resolvedModules`; every existing caller (`validate`, `plan`, `apply`, `compile`, `inspect`, `graph`, `status`-adjacent code, `drift`, `rollback`, `destroy`, `import`) continues to work unmodified since the new field is additive. No existing diagnostic code, exit code, or command flag changes meaning.
